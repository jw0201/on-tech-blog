서비스가 커지면 하나의 요청 안에서 데이터베이스를 바꾸고 이벤트도 발행해야 하는 일이 많아집니다. 주문이 생성되면 결제 준비 이벤트를 보내고, 회원이 가입하면 웰컴 메일 작업을 큐에 넣는 식입니다. 문제는 DB 커밋과 메시지 발행이 서로 다른 시스템이라는 점입니다.

DB 저장은 성공했지만 메시지 발행이 실패하면 downstream 시스템은 변화를 모릅니다. 반대로 메시지는 나갔지만 DB 트랜잭션이 롤백되면 존재하지 않는 데이터를 기준으로 후속 작업이 시작됩니다.

## DB 변경과 이벤트 기록을 같은 트랜잭션에 넣는다

Transactional outbox는 이벤트를 메시지 브로커에 바로 보내지 않고, 먼저 같은 데이터베이스 트랜잭션 안에서 outbox 테이블에 기록합니다.

```text
begin
  insert into orders ...
  insert into outbox_events ...
commit
```

이렇게 하면 주문 저장과 이벤트 기록은 같이 성공하거나 같이 실패합니다. 요청 처리 경로에서는 외부 브로커 상태에 덜 의존하게 됩니다.

## 별도 퍼블리셔가 outbox를 비운다

커밋된 outbox 이벤트는 백그라운드 퍼블리셔가 읽어서 Kafka, SQS, RabbitMQ 같은 메시징 시스템으로 발행합니다. 발행에 성공하면 `published_at`을 채우거나 상태를 바꿉니다.

```text
id
event_type
aggregate_id
payload
created_at
published_at
retry_count
```

퍼블리셔는 실패해도 다시 시도할 수 있어야 합니다. 네트워크 오류나 브로커 점검은 운영에서 자주 일어나는 정상적인 실패입니다.

## 소비자는 중복을 허용해야 한다

Outbox는 유실 가능성을 크게 낮추지만, 중복 발행 가능성은 남습니다. 퍼블리셔가 브로커에 메시지를 보낸 직후 `published_at` 저장에 실패하면 같은 이벤트를 다시 보낼 수 있습니다.

그래서 이벤트에는 고유한 `event_id`가 있어야 하고, 소비자는 이미 처리한 이벤트를 다시 받았을 때 안전하게 무시할 수 있어야 합니다. 적어도 돈, 재고, 권한처럼 중요한 작업에서는 소비자 쪽 멱등성 저장소가 필요합니다.

## 처리 지연을 관측한다

Outbox 테이블이 쌓이기만 하면 시스템은 조용히 낡은 상태가 됩니다. 이벤트 발행 지연은 반드시 지표로 봐야 합니다.

- 가장 오래된 미발행 이벤트 나이
- 미발행 이벤트 개수
- 발행 실패율
- 재시도 횟수 분포
- dead letter로 이동한 이벤트 수

특히 "가장 오래된 미발행 이벤트 나이"는 사용자 영향과 연결하기 좋은 지표입니다.

## 체크리스트

- 도메인 변경과 outbox 기록을 같은 DB 트랜잭션에 넣는다.
- 퍼블리셔는 실패를 재시도하고 상태를 기록한다.
- 이벤트에는 고유 `event_id`를 넣는다.
- 소비자는 중복 이벤트를 안전하게 처리한다.
- outbox 적체와 발행 지연을 알림으로 연결한다.

Transactional outbox는 분산 트랜잭션을 마법처럼 해결하지는 않습니다. 하지만 가장 흔한 실패 지점인 "DB는 바뀌었는데 이벤트가 사라지는 문제"를 단순하고 운영 가능한 방식으로 줄여줍니다.
