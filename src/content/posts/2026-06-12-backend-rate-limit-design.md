Rate Limit은 단순히 요청을 막는 기능이 아닙니다. 정상 사용자의 실수를 흡수하고, 과도한 자동화를 제어하며, 장애 전파를 줄이는 운영 장치입니다. 너무 느슨하면 시스템이 흔들리고, 너무 엄격하면 정상 사용자가 서비스가 고장났다고 느낍니다.

좋은 제한 정책은 "누구를", "무엇을 기준으로", "얼마나 오래", "어떻게 회복하게 할지"를 명확히 답해야 합니다.

## 보호하려는 자원을 먼저 정한다

요청 수만 보고 제한을 걸면 중요한 차이를 놓칩니다. 가벼운 조회 API 100회와 외부 결제사 호출 100회는 비용이 다릅니다. Rate Limit은 보호하려는 자원에 맞춰 설계해야 합니다.

- 로그인 시도: 계정 탈취와 credential stuffing 방지
- 검색 API: 비싼 DB 쿼리와 인덱스 부하 제어
- 파일 업로드: 저장소 비용과 바이러스 검사 부하 제어
- 이메일 발송: 스팸 악용과 외부 발송 한도 보호
- AI 추론 API: 토큰 비용과 큐 대기 시간 제어

같은 사용자라도 엔드포인트별 비용이 다르면 제한도 달라져야 합니다.

## 식별 키를 조합한다

IP만으로 제한하면 사무실, 학교, 통신사 NAT 뒤의 여러 사용자가 함께 묶입니다. 사용자 ID만으로 제한하면 로그인 전 공격을 막기 어렵습니다. 보통은 상황에 따라 여러 키를 조합합니다.

```text
login: ip + email
authenticated_api: user_id + endpoint
public_search: ip + user_agent + route
tenant_api: tenant_id + user_id + endpoint
```

멀티테넌트 서비스에서는 `tenant_id` 단위 제한도 필요합니다. 한 조직의 자동화가 전체 시스템을 잠식하지 않게 하려면 사용자 제한과 조직 제한을 함께 둬야 합니다.

## 응답은 회복 가능해야 한다

제한에 걸렸을 때 `429 Too Many Requests`만 반환하면 클라이언트는 언제 다시 시도해야 할지 알 수 없습니다. 서버는 가능한 한 회복 정보를 함께 제공해야 합니다.

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 30
RateLimit-Limit: 100
RateLimit-Remaining: 0
RateLimit-Reset: 1718000000
```

API 클라이언트는 이 값을 사용해 재시도 간격을 조정할 수 있습니다. UI에서는 무작정 실패 메시지를 보여주기보다 "잠시 후 다시 시도해 주세요"처럼 사용자가 다음 행동을 이해할 수 있게 해야 합니다.

## 고정 윈도우의 경계 문제를 주의한다

가장 단순한 방식은 1분에 100회처럼 고정된 시간 창을 두는 것입니다. 구현은 쉽지만 경계 시점에 요청이 몰릴 수 있습니다. 예를 들어 12:00:59에 100회, 12:01:00에 100회를 보내면 사실상 1초 안에 200회가 들어옵니다.

트래픽이 크거나 비용이 비싼 API라면 sliding window, token bucket, leaky bucket 같은 알고리즘을 검토해야 합니다. 특히 token bucket은 일정한 평균 요청량을 유지하면서 짧은 burst를 허용할 수 있어 사용자 경험과 보호 사이의 균형이 좋습니다.

## 운영자가 풀 수 있어야 한다

Rate Limit은 운영 중 예외가 자주 필요합니다. 파트너 연동, 내부 배치, 마이그레이션, 고객사별 계약 한도처럼 일괄 규칙만으로 처리하기 어려운 경우가 생깁니다.

정책은 코드에 하드코딩하기보다 설정으로 관리하는 편이 좋습니다. 다만 예외를 무제한으로 늘리면 정책이 무너집니다. 예외에는 만료일, 사유, 담당자, 적용 범위를 남겨야 합니다.

## 체크리스트

- 보호할 자원과 비용을 먼저 정의한다.
- IP, 사용자, 테넌트, 엔드포인트 기준을 조합한다.
- `429` 응답에 재시도 정보를 포함한다.
- 고정 윈도우의 경계 burst를 고려한다.
- 예외 정책에는 만료일과 사유를 둔다.
- 제한 이벤트를 로그와 대시보드로 관찰한다.

Rate Limit은 보안 기능이면서 신뢰성 기능입니다. 사용자를 벌주는 장치가 아니라, 시스템이 예측 가능한 방식으로 버티게 만드는 계약으로 설계해야 합니다.
