Optimistic UI는 서버 응답을 기다리지 않고 화면을 먼저 바꾸는 패턴입니다. 좋아요, 체크박스, 칸반 카드 이동처럼 사용자가 즉시 피드백을 기대하는 기능에서 큰 효과가 있습니다. 하지만 단순히 "먼저 바꾸고 실패하면 되돌린다"로 끝내면 실제 제품에서는 금방 복잡해집니다.

좋은 Optimistic UI는 성공 경로보다 실패 경로를 더 구체적으로 설계합니다. 네트워크 지연, 중복 클릭, 권한 만료, 서버 검증 실패, 다른 기기에서의 변경 같은 상황이 모두 화면 상태에 영향을 주기 때문입니다.

## 낙관 상태를 별도로 표시한다

서버에 저장된 상태와 사용자가 방금 요청한 상태를 같은 값으로만 다루면 디버깅이 어렵습니다. UI 내부에서는 최소한 다음 상태를 구분하는 편이 좋습니다.

- `synced`: 서버와 화면이 일치함
- `pending`: 화면은 바뀌었지만 서버 확인 전
- `failed`: 요청 실패로 사용자 조치가 필요함
- `conflicted`: 서버의 최신 상태와 로컬 요청이 충돌함

예를 들어 할 일 완료 버튼을 누르면 체크박스를 즉시 켜되, 해당 항목에는 `pending` 플래그를 붙입니다. 이 플래그는 작은 스피너, 흐린 텍스트, 재시도 버튼처럼 UI 표현으로 이어질 수 있습니다.

## 롤백보다 보정이 나은 경우가 있다

모든 실패를 원래 상태로 되돌리는 것은 자연스럽지 않을 때가 많습니다. 사용자가 댓글을 작성했는데 네트워크가 끊겼다면 댓글을 즉시 없애기보다 "전송 실패" 상태로 남겨두고 재시도할 수 있게 하는 편이 좋습니다.

반대로 좋아요 토글처럼 입력 비용이 낮고 상태가 작다면 실패 시 원래 값으로 되돌리는 것이 단순합니다. 기능마다 복구 전략을 다르게 잡아야 합니다.

```text
좋아요: 실패하면 이전 값으로 롤백
댓글 작성: 실패한 초안을 보존하고 재시도 제공
결제 요청: 낙관 업데이트 금지, 서버 확정 후 반영
권한 변경: pending 상태를 명확히 표시하고 감사 로그 기준으로 확정
```

특히 돈, 권한, 재고처럼 결과가 민감한 기능은 Optimistic UI를 피하거나 매우 제한적으로 써야 합니다.

## 중복 요청을 식별한다

사용자가 같은 버튼을 여러 번 누르거나 브라우저가 재시도하면 서버에는 중복 요청이 들어갈 수 있습니다. 클라이언트에서 버튼을 잠그는 것만으로는 부족합니다. 요청마다 `clientMutationId` 같은 식별자를 붙이면 서버와 클라이언트가 같은 변경을 하나의 작업으로 추적할 수 있습니다.

```json
{
  "clientMutationId": "todo-42-complete-20260614T045900Z",
  "todoId": "42",
  "completed": true
}
```

이 값은 로그에서도 유용합니다. 사용자가 "체크했는데 다시 풀렸다"고 말할 때 특정 변경 요청이 서버에서 어떻게 처리됐는지 추적할 수 있습니다.

## 서버 응답으로 최종 상태를 덮어쓴다

낙관 업데이트 후 성공 응답이 오면 로컬에서 추정한 값을 그대로 믿지 말고 서버가 반환한 최종 상태로 교체해야 합니다. 서버는 정렬 순서, 수정 시간, 권한 필드, 계산된 카운트 등을 함께 바꿀 수 있습니다.

목록 화면에서는 특히 중요합니다. 카드를 다른 컬럼으로 옮겼을 때 서버가 실제 우선순위나 위치 값을 다시 계산했다면, 응답 값을 기준으로 캐시를 정리해야 다음 새로고침에서 화면이 흔들리지 않습니다.

## 사용자에게 실패를 숨기지 않는다

Optimistic UI의 목적은 실패를 감추는 것이 아니라 정상 경로를 빠르게 느끼게 하는 것입니다. 실패했는데 조용히 원복되면 사용자는 본인이 잘못 눌렀는지 서비스가 고장났는지 알 수 없습니다.

좋은 실패 메시지는 다음 행동을 포함합니다.

- "저장하지 못했습니다. 다시 시도"
- "권한이 변경되어 완료 처리할 수 없습니다."
- "오프라인 상태입니다. 연결되면 자동으로 전송합니다."

실패가 잦은 기능이라면 화면 메시지뿐 아니라 이벤트 로그와 모니터링도 필요합니다. Optimistic UI 실패율은 프론트엔드 품질 지표로 볼 수 있습니다.

## 체크리스트

- 서버 상태와 낙관 상태를 내부 모델에서 구분한다.
- 기능별로 롤백, 보존, 재시도, 낙관 금지 전략을 정한다.
- 요청마다 클라이언트 식별자를 붙여 중복과 추적을 처리한다.
- 성공 응답 후에는 서버의 최종 상태로 캐시를 정리한다.
- 실패는 사용자에게 회복 가능한 방식으로 보여준다.

Optimistic UI는 속도 트릭이 아니라 분산 상태 설계입니다. 사용자가 빠르게 느끼는 화면 뒤에는 실패를 정직하게 다루는 구조가 있어야 합니다.
