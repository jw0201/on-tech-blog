롤백은 배포 전략의 부록이 아닙니다. 배포가 실패할 수 있다는 사실을 인정하고, 실패했을 때 복구 시간을 줄이는 핵심 절차입니다. 그런데 많은 팀의 롤백 문서는 "문제가 있으면 이전 버전으로 되돌린다" 정도에서 멈춥니다. 장애 중에는 그런 문장이 거의 도움이 되지 않습니다.

좋은 롤백 런북은 누가 봐도 같은 행동을 할 수 있어야 합니다. 특히 새벽에 잠에서 깬 온콜 담당자가 읽어도 실행 가능한 수준이어야 합니다.

## 롤백 판단 기준을 숫자로 둔다

먼저 언제 롤백할지 정해야 합니다. "심각한 문제가 생기면"이라는 기준은 사람마다 다르게 해석됩니다. 가능한 한 관측 가능한 신호로 정의합니다.

- 배포 후 10분 안에 5xx 비율이 평소 대비 3배 이상 증가
- 결제 성공률이 2분 연속 95% 미만
- 주요 API p95 지연 시간이 1초를 초과
- 에러 예산 소모 속도가 평소 대비 5배 이상
- 고객 문의 또는 내부 알림에서 동일 증상이 3건 이상 확인

모든 기준이 자동화될 필요는 없지만, 롤백을 망설이는 시간을 줄여야 합니다.

## 되돌릴 대상과 되돌리지 못할 대상을 구분한다

애플리케이션 이미지는 쉽게 롤백할 수 있어도 데이터베이스 마이그레이션은 그렇지 않을 수 있습니다. 런북에는 되돌릴 수 있는 것과 없는 것을 분리해서 적어야 합니다.

```text
롤백 가능:
- 애플리케이션 컨테이너 이미지
- feature flag
- CDN 설정
- worker 배포 버전

주의 필요:
- destructive DB migration
- 외부 API 계약 변경
- 메시지 큐 payload 형식 변경
- 캐시 키 포맷 변경
```

되돌릴 수 없는 변경은 배포 전에 forward-compatible하게 설계해야 합니다. 예를 들어 컬럼 삭제는 바로 하지 않고, 코드가 새 컬럼과 이전 컬럼을 모두 처리하는 기간을 둡니다.

## 명령어는 복사해서 실행 가능해야 한다

런북의 명령어는 추상적이면 안 됩니다. 환경 변수, 네임스페이스, 서비스 이름, 확인 명령까지 포함해야 합니다.

```bash
kubectl -n production rollout undo deployment/api
kubectl -n production rollout status deployment/api
kubectl -n production get pods -l app=api
```

클라우드 콘솔에서 해야 하는 작업도 화면 이름과 메뉴 경로를 적습니다. 장애 중에는 기억력보다 문서의 정확성이 더 믿을 만합니다.

## 롤백 후 확인 항목을 둔다

이전 버전으로 되돌렸다고 끝이 아닙니다. 실제로 사용자 영향이 줄었는지 확인해야 합니다.

- 새 버전 pod 또는 instance가 더 이상 트래픽을 받지 않는가
- 오류율과 지연 시간이 정상 범위로 돌아왔는가
- 큐 적체가 줄어들고 있는가
- 외부 연동 실패가 회복되고 있는가
- 고객이 보는 주요 경로가 수동 테스트로 통과하는가

이 확인 항목이 없으면 롤백이 성공했다고 착각한 채 다른 원인을 놓칠 수 있습니다.

## 정기적으로 리허설한다

롤백 런북은 실제로 실행해 보기 전까지 믿을 수 없습니다. 스테이징 환경에서 정기적으로 롤백 리허설을 하거나, 낮은 위험의 서비스에서 운영 리허설을 해 보는 것이 좋습니다.

리허설 후에는 문서와 자동화를 바로 고칩니다. 명령어가 오래되었거나 권한이 부족하거나 대시보드 링크가 깨져 있다면, 장애가 아닌 날 발견한 것이 다행입니다.

## 체크리스트

- 롤백 판단 기준을 관측 가능한 지표로 적는다.
- 되돌릴 수 있는 변경과 주의가 필요한 변경을 구분한다.
- 명령어와 콘솔 경로를 복사 실행 가능한 수준으로 쓴다.
- 롤백 후 정상화 확인 항목을 둔다.
- 스테이징이나 저위험 서비스에서 정기적으로 리허설한다.
- 리허설 결과를 런북에 즉시 반영한다.

롤백 런북의 목표는 멋진 문서가 아니라 빠른 복구입니다. 장애 순간에 팀이 토론보다 실행에 집중할 수 있다면, 그 런북은 이미 충분히 가치가 있습니다.
