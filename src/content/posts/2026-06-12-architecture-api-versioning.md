API는 한 번 공개되면 혼자 바꾸기 어렵습니다. 웹 프론트엔드, 모바일 앱, 파트너 서버, 내부 배치가 같은 API에 의존할 수 있습니다. 서버 코드는 배포하면 바뀌지만, 클라이언트는 사용자의 업데이트 속도와 외부 파트너 일정에 묶입니다.

그래서 API 버저닝은 URL에 `v1`을 붙이는 문제를 넘어 호환성을 관리하는 방식입니다.

## 깨지는 변경을 정의한다

먼저 팀 안에서 breaking change의 기준을 합의해야 합니다. 다음 변경은 대체로 클라이언트를 깨뜨릴 수 있습니다.

- 필수 응답 필드 제거
- 필드 타입 변경
- enum 값의 의미 변경
- 필수 요청 필드 추가
- 에러 코드 변경
- pagination 형식 변경
- 인증 방식 변경

반대로 응답에 optional 필드를 추가하는 것은 보통 호환 가능한 변경입니다. 다만 엄격한 스키마 검증을 하는 클라이언트가 있다면 새 필드 추가도 문제가 될 수 있으므로 파트너 API에서는 더 조심해야 합니다.

## 확장 가능한 응답을 설계한다

처음부터 확장 여지를 두면 버전 증가를 줄일 수 있습니다. enum은 알 수 없는 값을 받을 수 있다는 전제로 클라이언트를 설계하고, 응답 객체에는 optional 필드를 추가할 수 있게 합니다.

```json
{
  "id": "ord_123",
  "status": "paid",
  "amount": 39000,
  "currency": "KRW",
  "metadata": {}
}
```

`metadata` 같은 자유 영역을 남용하면 계약이 흐려질 수 있지만, 외부 연동에서 작은 확장을 흡수하는 데는 유용합니다. 중요한 필드는 문서화된 정식 필드로 승격하는 기준도 있어야 합니다.

## 버전 위치를 일관되게 정한다

버전은 URL, 헤더, 미디어 타입, 날짜 기반 버전 등 여러 방식으로 표현할 수 있습니다.

```http
GET /v1/orders
Accept: application/vnd.example.v1+json
X-API-Version: 2026-06-12
```

일반적인 REST API에서는 URL 버전이 이해하기 쉽고 운영 도구와도 잘 맞습니다. 헤더 버전은 URL을 깔끔하게 유지할 수 있지만 디버깅과 캐시 설정이 더 복잡할 수 있습니다. 무엇을 선택하든 팀과 사용자에게 일관되게 적용하는 것이 중요합니다.

## 새 버전은 복사가 아니라 분리된 계약이다

`v2`를 만들 때 기존 컨트롤러를 통째로 복사하면 빠르지만, 이후 버그 수정이 두 곳에 필요해질 수 있습니다. 반대로 하나의 코드에서 조건문으로 모든 버전을 처리하면 복잡도가 올라갑니다.

실무에서는 외부 계약 계층과 내부 도메인 로직을 나누는 방식이 좋습니다.

```text
v1 request -> v1 adapter -> domain service -> v1 response
v2 request -> v2 adapter -> domain service -> v2 response
```

도메인 로직은 공유하되, 요청과 응답 형태는 버전별 어댑터에서 관리합니다. 이렇게 하면 API 계약 변경이 내부 모델 전체를 흔들지 않습니다.

## 폐기 정책을 문서화한다

오래된 버전을 영원히 지원하면 유지보수 비용이 커집니다. 새 버전을 내는 것만큼 이전 버전을 어떻게 종료할지도 중요합니다.

폐기 정책에는 다음이 포함되어야 합니다.

- 지원 종료 예고 기간
- 영향을 받는 엔드포인트
- 대체 버전과 마이그레이션 가이드
- 호출량이 남아 있는 고객 확인 방법
- 종료 후 응답 코드와 메시지

파트너 API라면 이메일, 대시보드 알림, 응답 헤더를 함께 사용하는 것이 좋습니다.

## 체크리스트

- 팀 차원의 breaking change 기준을 문서화한다.
- 응답은 optional 필드 추가와 unknown enum에 견디게 설계한다.
- 버전 위치를 정하고 모든 API에 일관되게 적용한다.
- 버전별 어댑터로 외부 계약과 내부 도메인을 분리한다.
- 새 버전 출시와 함께 이전 버전 폐기 계획을 만든다.
- 실제 호출량을 보고 폐기 일정을 조정한다.

API 버저닝의 목표는 버전을 많이 만드는 것이 아닙니다. 필요한 변경을 하면서도 기존 클라이언트가 예측 가능하게 움직이도록 신뢰를 지키는 것입니다.
