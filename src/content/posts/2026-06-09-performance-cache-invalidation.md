캐시는 빠르게 만드는 기술이지만 동시에 오래된 값을 보여줄 위험을 만든다. 그래서 캐시를 붙일 때는 "어디에 저장할까"보다 "언제 틀려도 되는가"를 먼저 정해야 합니다. 무효화 기준 없이 캐시를 추가하면 성능 문제는 줄어도 데이터 신뢰 문제가 생깁니다.

좋은 캐시 설계는 데이터의 신선도 요구사항을 명확히 하는 것에서 시작합니다.

## 데이터별 허용 지연을 정한다

모든 데이터가 실시간일 필요는 없습니다. 공지 목록은 몇 분 늦어도 괜찮을 수 있고, 재고나 권한은 몇 초의 지연도 문제가 될 수 있습니다.

먼저 데이터 유형별로 허용 가능한 stale 시간을 정합니다.

- 상품 설명: 수 분에서 수 시간
- 인기 검색어: 수 분
- 사용자 권한: 가능한 짧게
- 결제 상태: 캐시보다 원본 조회 우선
- 피드 카운트: 약간의 지연 허용

이 기준이 있어야 TTL을 감으로 정하지 않습니다.

## 캐시 키에는 변하는 조건을 모두 넣는다

캐시 키가 너무 단순하면 다른 조건의 응답이 섞입니다. 사용자, locale, 권한, 실험군, API 버전이 응답에 영향을 준다면 key에도 반영해야 합니다.

```text
product:{productId}:locale:{locale}:v2
user-menu:{userId}:roleVersion:{roleVersion}
```

반대로 불필요하게 세부적인 값을 모두 넣으면 hit rate가 떨어집니다. key 설계는 정확성과 재사용률 사이의 균형입니다.

## TTL은 안전망이지 유일한 전략이 아니다

TTL만으로 운영하면 데이터 변경 직후 오래된 값이 계속 보일 수 있습니다. 변경 빈도가 낮고 stale 허용 시간이 긴 데이터라면 충분하지만, 사용자가 바로 결과를 기대하는 기능에는 부족합니다.

중요한 데이터는 쓰기 경로에서 관련 캐시를 지우거나, 도메인 이벤트를 발행해 무효화 worker가 처리하게 할 수 있습니다. 다만 이벤트 기반 무효화도 실패할 수 있으므로 짧은 TTL을 마지막 안전망으로 두는 편이 좋습니다.

## Stale-while-revalidate를 활용한다

읽기 트래픽이 많은 데이터는 캐시 만료 순간에 요청이 몰릴 수 있습니다. 이때 stale-while-revalidate 전략을 쓰면 사용자는 잠깐 오래된 값을 받고, 백그라운드에서 새 값을 채웁니다.

이 전략은 목록, 추천, 통계처럼 약간의 지연을 허용하는 데이터에 적합합니다. 권한, 잔액, 결제 상태처럼 정확성이 중요한 데이터에는 조심해야 합니다.

## 체크리스트

- 데이터별로 허용 가능한 stale 시간을 정한다.
- 응답을 바꾸는 조건은 캐시 키에 포함한다.
- TTL은 안전망으로 두고 필요하면 명시적 무효화를 추가한다.
- 이벤트 기반 무효화 실패를 고려한다.
- stale-while-revalidate는 정확성 요구가 낮은 데이터에 쓴다.

캐시는 단순한 성능 최적화가 아니라 데이터 계약입니다. 사용자가 얼마나 오래된 정보를 봐도 되는지 합의하지 않으면, 빠른 응답은 곧 빠른 오답이 될 수 있습니다.
