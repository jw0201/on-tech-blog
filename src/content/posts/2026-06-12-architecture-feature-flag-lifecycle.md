Feature Flag는 배포와 출시를 분리해 줍니다. 코드는 미리 배포하고, 기능 노출은 특정 사용자나 조직에만 켤 수 있습니다. 장애가 나면 플래그를 끄는 방식으로 빠르게 복구할 수도 있습니다. 하지만 플래그가 늘어나면 코드가 복잡해지고, 오래된 조건문이 제품 동작을 이해하기 어렵게 만듭니다.

Feature Flag는 만들 때보다 지울 때 더 중요합니다. 생명주기를 설계하지 않으면 플래그 시스템은 안전 장치가 아니라 코드베이스의 안개가 됩니다.

## 플래그 목적을 먼저 정한다

모든 플래그가 같은 성격은 아닙니다. 목적에 따라 운영 방식과 정리 시점이 달라집니다.

- release flag: 새 기능을 점진적으로 출시
- experiment flag: A/B 테스트와 실험
- ops flag: 장애 시 기능을 끄는 운영 스위치
- permission flag: 요금제나 권한에 따른 기능 노출
- migration flag: 데이터나 인프라 전환 단계 제어

release flag는 출시가 끝나면 제거해야 합니다. 반면 permission flag는 제품 정책의 일부라 오래 유지될 수 있습니다. 플래그 이름과 설명에 목적이 드러나야 나중에 지워도 되는지 판단할 수 있습니다.

## 기본값은 안전하게 둔다

플래그 서비스가 장애 나거나 SDK 초기화가 늦어질 때 어떤 값으로 동작할지 정해야 합니다. 일반적으로 새 기능 release flag는 `false`가 안전합니다. 하지만 보안 차단이나 장애 완화용 ops flag는 기본값이 `true`여야 할 수도 있습니다.

```ts
const enabled = flags.get("new-checkout", { defaultValue: false });
```

기본값을 호출부마다 다르게 두면 같은 플래그가 화면마다 다르게 동작할 수 있습니다. 플래그 정의를 중앙에 모으고 기본값과 소유자를 함께 관리하는 편이 좋습니다.

## 조건문 범위를 좁힌다

플래그 조건문이 여러 계층에 흩어지면 조합을 이해하기 어렵습니다. 가능하면 분기 지점을 사용자 경험이나 서비스 경계에 가깝게 둡니다.

```ts
if (flags.newCheckout) {
  return <NewCheckoutPage />;
}

return <LegacyCheckoutPage />;
```

반대로 컴포넌트 내부 곳곳에 `if (newCheckout)`가 퍼지면 제거 비용이 커집니다. 백엔드에서도 새로운 처리 경로와 기존 처리 경로를 명확히 분리하고, 공통 부작용은 중복 실행되지 않게 주의해야 합니다.

## 관측 가능성을 붙인다

플래그를 켰는지 껐는지 모르면 장애 분석이 어려워집니다. 주요 요청 로그와 이벤트에는 플래그 변형값을 남기는 것이 좋습니다.

```json
{
  "route": "/checkout",
  "user_id": "u_123",
  "flag_new_checkout": "enabled",
  "latency_ms": 240
}
```

실험용 플래그라면 분석 이벤트에도 variant를 포함해야 합니다. 그래야 전환율이나 오류율을 플래그 상태별로 비교할 수 있습니다.

## 제거 티켓을 같이 만든다

플래그를 만들 때 제거 조건을 함께 정합니다. "100% 출시 후 2주 동안 주요 지표 안정", "실험 종료 후 의사결정 완료", "마이그레이션 검증 완료 후 이전 코드 삭제"처럼 구체적인 조건이 필요합니다.

코드 리뷰에서는 새 플래그가 추가될 때 다음 정보를 확인합니다.

- 소유자
- 목적
- 기본값
- 예상 제거일
- 제거 조건
- 관련 대시보드 또는 지표

이 정보가 없으면 플래그는 높은 확률로 영구 조건문이 됩니다.

## 체크리스트

- 플래그 목적을 release, experiment, ops, permission, migration 중 하나로 분류한다.
- 기본값과 장애 시 동작을 중앙에서 정의한다.
- 조건문은 가능한 한 좁고 명확한 경계에 둔다.
- 로그와 분석 이벤트에 주요 플래그 값을 포함한다.
- 플래그 생성 시 제거 조건과 소유자를 함께 남긴다.
- 정기적으로 오래된 플래그를 삭제하는 시간을 둔다.

Feature Flag는 빠른 배포를 위한 도구입니다. 하지만 정리되지 않은 플래그는 속도를 빼앗습니다. 생성부터 제거까지 생명주기를 관리해야 플래그가 계속 팀을 돕습니다.
