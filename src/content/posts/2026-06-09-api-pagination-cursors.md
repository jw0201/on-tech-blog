목록 API는 처음에는 단순해 보입니다. `page=1&limit=20`으로 시작해도 작은 데이터에서는 잘 동작합니다. 하지만 데이터가 많아지고 새 row가 계속 추가되면 offset 기반 페이지네이션은 느려지고, 사용자는 중복되거나 누락된 항목을 보게 됩니다.

커서 기반 페이지네이션은 "몇 번째 페이지인가" 대신 "마지막으로 본 항목 이후를 달라"는 방식입니다.

## 안정적인 정렬 기준이 필요하다

커서는 정렬 기준이 안정적일 때만 제대로 동작합니다. `created_at desc`만으로 정렬하면 같은 시각에 생성된 항목 사이의 순서가 흔들릴 수 있습니다. 보통은 고유한 보조 키를 함께 씁니다.

```sql
order by created_at desc, id desc
```

다음 페이지를 가져올 때는 마지막 항목의 `created_at`과 `id`를 기준으로 이어서 조회합니다.

```sql
where (created_at, id) < (:lastCreatedAt, :lastId)
order by created_at desc, id desc
limit 20
```

이 방식은 새 데이터가 앞에 추가되어도 사용자가 보고 있던 흐름을 비교적 안정적으로 유지합니다.

## 커서에는 내부 구현을 그대로 노출하지 않는다

커서를 단순히 `id`로만 만들 수도 있지만, 정렬 조건이 바뀌거나 복합 키가 필요해지면 한계가 생깁니다. 실무에서는 필요한 값을 JSON으로 묶고 base64로 인코딩하는 방식을 자주 씁니다.

```json
{
  "createdAt": "2026-06-09T08:00:00Z",
  "id": "post_123"
}
```

커서에 민감 정보나 권한 정보를 넣으면 안 됩니다. 사용자가 조작할 수 있는 입력으로 보고 검증해야 합니다. 필요하면 서명을 붙여 변조를 막습니다.

## 필터가 바뀌면 커서도 무효다

커서는 특정 쿼리 조건에서만 의미가 있습니다. 필터, 검색어, 정렬이 바뀌었는데 이전 커서를 재사용하면 엉뚱한 결과가 나올 수 있습니다.

API는 커서가 어떤 조건에서 만들어졌는지 확인하거나, 클라이언트가 조건 변경 시 커서를 버리도록 계약을 명확히 해야 합니다. 안전하게 하려면 커서 payload에 query fingerprint를 포함할 수 있습니다.

## total count에 집착하지 않는다

대용량 목록에서 매 요청마다 정확한 total count를 계산하면 성능 비용이 큽니다. 무한 스크롤 UX라면 보통 `hasNextPage`와 `nextCursor`만으로 충분합니다.

```json
{
  "items": [],
  "pageInfo": {
    "hasNextPage": true,
    "nextCursor": "..."
  }
}
```

정확한 전체 개수가 정말 필요하다면 별도 집계 테이블이나 근사값을 고려합니다.

## 체크리스트

- 정렬 기준은 고유하고 안정적으로 만든다.
- 커서는 마지막 항목의 정렬 키를 담는다.
- 커서 입력은 변조 가능하다고 보고 검증한다.
- 필터와 정렬이 바뀌면 기존 커서를 버린다.
- 무한 스크롤에는 total count보다 `hasNextPage`가 더 실용적이다.

커서 기반 페이지네이션은 단순히 성능 최적화가 아닙니다. 계속 변하는 데이터 안에서 사용자가 안정적인 목록 경험을 하게 만드는 API 계약입니다.
