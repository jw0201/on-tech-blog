PR 템플릿은 형식적인 체크박스가 되기 쉽습니다. 하지만 잘 만든 템플릿은 리뷰어가 코드를 읽기 전에 필요한 맥락을 빠르게 파악하게 해 줍니다. 리뷰 시간이 줄고, 질문이 줄고, 중요한 설계 판단에 더 많은 에너지를 쓸 수 있습니다.

좋은 PR 설명은 "무엇을 바꿨는가"보다 "왜 이렇게 바꿨는가"를 설명합니다. diff는 무엇을 바꿨는지 이미 보여주기 때문입니다.

## 변경 이유를 먼저 쓴다

리뷰어는 코드보다 문제 상황을 먼저 이해해야 합니다. 버그 수정이라면 어떤 증상이 있었는지, 기능 추가라면 어떤 사용자 흐름을 지원하는지, 리팩터링이라면 어떤 위험을 줄이는지 적습니다.

```md
## 배경

주문 목록에서 필터를 바꾼 뒤 뒤로 가기를 누르면 이전 필터가 복원되지 않았다.
필터 상태를 URL 쿼리로 옮겨 새로고침과 공유 링크에서도 같은 결과가 보이게 한다.
```

이 정도만 있어도 리뷰어는 변경 방향을 이해하고 diff를 읽을 수 있습니다.

## 리뷰 포인트를 지정한다

모든 코드를 같은 깊이로 봐 달라고 하면 리뷰어는 어디에 집중해야 할지 모릅니다. 작성자가 걱정하는 부분을 직접 알려주는 편이 좋습니다.

```md
## 리뷰 포인트

- 쿼리 파라미터 기본값 처리 방식이 기존 검색 화면과 일관적인지
- 잘못된 `page` 값이 들어왔을 때 1로 보정하는 정책이 괜찮은지
- `useOrderFilters` 훅 이름과 책임 범위가 적절한지
```

리뷰 포인트는 방어가 아니라 초대입니다. 설계 판단이 필요한 곳을 드러내면 리뷰 품질이 올라갑니다.

## 테스트 결과는 재현 가능하게 적는다

"테스트 완료"는 정보가 거의 없습니다. 어떤 명령을 실행했고, 어떤 수동 시나리오를 확인했는지 적어야 합니다.

```md
## 검증

- `npm test -- order-filters`
- `npm run build`
- 주문 목록에서 상태 필터 변경 후 새로고침
- 잘못된 `?page=abc` 접근 시 1페이지로 보정 확인
```

테스트를 못 했다면 이유를 적습니다. 숨기는 것보다 남기는 것이 리뷰어에게 훨씬 낫습니다.

## 스크린샷은 변화가 있을 때만

UI 변경이 있다면 스크린샷이나 짧은 녹화가 큰 도움이 됩니다. 하지만 백엔드 내부 변경이나 테스트 리팩터링에도 무조건 스크린샷 항목을 요구하면 템플릿이 소음이 됩니다.

템플릿에는 "UI 변경 시"처럼 조건을 붙이는 편이 좋습니다. 모바일과 데스크톱 차이가 중요한 화면이라면 둘 다 첨부하도록 안내합니다.

## 체크박스는 적게 둔다

체크박스가 많아지면 사람들은 읽지 않고 모두 체크합니다. 정말 중요한 팀 규칙만 남기는 것이 좋습니다.

좋은 체크박스는 다음처럼 행동이 명확합니다.

```md
- [ ] 사용자 입력을 다루는 경우 검증 실패 경로를 확인했다.
- [ ] DB 마이그레이션이 있는 경우 롤백 또는 forward-compatible 전략을 적었다.
- [ ] 외부 API 계약 변경이 있는 경우 소비자 영향을 확인했다.
```

반대로 "코드를 깨끗하게 작성했다" 같은 항목은 검증하기 어렵고 거의 도움이 되지 않습니다.

## 예시 템플릿

```md
## 배경

## 변경 사항

## 리뷰 포인트

## 검증

## UI 변경 시 스크린샷

## 배포/운영 참고
```

짧지만 이 정도면 대부분의 PR에 필요한 맥락을 담을 수 있습니다. 팀 성격에 따라 보안, 마이그레이션, 성능 항목을 조건부로 추가하면 됩니다.

## 체크리스트

- PR 설명은 변경 이유와 문제 상황으로 시작한다.
- 작성자가 원하는 리뷰 포인트를 명시한다.
- 테스트 결과는 실행 명령과 수동 확인 시나리오로 적는다.
- UI 변경이 있을 때만 스크린샷 항목을 요구한다.
- 체크박스는 실제 행동을 바꾸는 항목만 남긴다.
- 템플릿은 주기적으로 줄이고 다듬는다.

PR 템플릿은 팀의 사고 과정을 정리하는 작은 도구입니다. 리뷰어가 맥락을 빠르게 잡을수록, 리뷰는 취향 싸움보다 더 나은 변경을 만드는 대화에 가까워집니다.
