간헐적으로 실패하는 테스트는 단순한 귀찮음이 아닙니다. 테스트 스위트에 대한 신뢰를 떨어뜨리고, 실제 회귀를 놓치게 만들며, 배포 시간을 늘립니다. 더 위험한 점은 팀이 실패를 "또 flaky겠지"라고 넘기기 시작하는 순간입니다.

Flaky test를 줄이려면 실패한 테스트를 무작정 재실행하는 대신 원인을 분류해야 합니다. 원인이 다르면 해결책도 다릅니다.

## 시간 의존 테스트

가장 흔한 유형은 시간에 의존하는 테스트입니다. 현재 시각, 타임존, 날짜 경계, 타이머가 테스트 결과에 영향을 줍니다.

```ts
expect(formatDueDate(new Date())).toBe("오늘");
```

이런 테스트는 실행 시각에 따라 달라질 수 있습니다. 해결책은 시간을 주입하거나 고정하는 것입니다. 함수가 내부에서 `Date.now()`를 직접 호출하기보다 clock 객체를 받게 하면 테스트가 안정됩니다.

```ts
formatDueDate(task, { now: new Date("2026-06-12T09:00:00+09:00") });
```

시간을 다루는 테스트는 타임존도 명시해야 합니다. 로컬에서는 통과하고 CI에서 실패하는 문제는 타임존 차이에서 자주 발생합니다.

## 비동기 대기 문제

프론트엔드 테스트와 E2E 테스트에서는 비동기 작업이 끝나기 전에 검증해서 실패하는 경우가 많습니다. 고정 sleep으로 해결하면 느리고 불안정합니다.

```ts
await page.waitForTimeout(1000);
```

가능하면 사용자가 보는 상태나 네트워크 조건을 기다려야 합니다.

```ts
await expect(page.getByRole("status")).toHaveText("저장됨");
```

테스트는 시간이 지난 것을 기다리는 게 아니라, 시스템이 특정 상태가 된 것을 기다려야 합니다.

## 공유 상태 오염

테스트 간 공유 상태도 흔한 원인입니다. DB 레코드, 로컬 스토리지, 캐시, 전역 mock, feature flag가 이전 테스트의 영향을 받을 수 있습니다.

이 유형은 단독 실행하면 통과하고 전체 실행하면 실패하는 특징이 있습니다. 해결하려면 테스트마다 독립된 데이터를 만들고, 실행 후 정리하거나 트랜잭션 롤백을 사용합니다. 병렬 실행 환경에서는 데이터 이름에 고유 suffix를 붙이는 것도 도움이 됩니다.

```text
user-email: test+{workerId}-{timestamp}@example.test
```

전역 mock은 테스트가 끝날 때 반드시 복구해야 합니다.

## 외부 의존성 흔들림

네트워크, 실제 결제 sandbox, 외부 API, 브라우저 다운로드처럼 외부 시스템에 의존하면 테스트가 불안정해집니다. 통합 테스트에서 일부 외부 의존성은 필요할 수 있지만, 대부분의 CI 테스트는 계약 기반 mock이나 local fake로 대체하는 편이 좋습니다.

외부 시스템을 사용하는 테스트는 별도 job으로 분리하고, 실패가 제품 코드의 회귀인지 환경 문제인지 구분할 수 있게 로그를 남깁니다.

## 격리보다 무시가 쉬워지는 순간을 경계한다

Flaky test에 `skip`을 붙이는 것은 마지막 수단이어야 합니다. 정말로 비활성화해야 한다면 이슈 번호, 담당자, 만료일을 같이 남깁니다.

```ts
test.skip("ISSUE-1234: payment webhook ordering is flaky, revisit by 2026-06-30", () => {
  // ...
});
```

만료일 없는 skip은 테스트 스위트의 부채로 굳어집니다.

## 체크리스트

- 실패를 재실행하기 전에 원인 유형을 기록한다.
- 현재 시각과 타임존은 테스트에서 고정한다.
- 고정 sleep 대신 관측 가능한 상태를 기다린다.
- 테스트 데이터와 전역 mock을 격리한다.
- 외부 의존성 테스트는 별도 job으로 분리한다.
- skip에는 이슈, 담당자, 만료일을 남긴다.

테스트는 빠른 피드백을 주기 위해 존재합니다. Flaky test를 꾸준히 분류하고 줄이면 CI는 배포를 막는 장애물이 아니라 팀이 믿고 달릴 수 있는 계기판이 됩니다.
