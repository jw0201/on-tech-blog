API 에러 응답은 단순한 실패 메시지가 아닙니다. 클라이언트가 다음 행동을 결정하고, 운영자가 장애를 분류하며, 고객지원팀이 사용자에게 설명하는 기준입니다. 그래서 에러 코드는 HTTP 상태 코드 몇 개를 감싸는 문자열이 아니라 팀 전체의 운영 언어가 되어야 합니다.

나쁜 에러 응답은 대부분 비슷합니다. 모든 실패가 `400 Bad Request`나 `500 Internal Server Error`로 뭉개지고, 메시지는 "something went wrong"입니다. 이런 응답은 개발 중에는 편하지만 제품이 커질수록 비용이 커집니다.

## HTTP 상태와 제품 코드를 분리한다

HTTP 상태 코드는 전송 계층의 큰 분류에 가깝습니다. 제품에서 필요한 세부 원인은 별도 코드로 표현하는 편이 좋습니다.

```json
{
  "error": {
    "code": "PAYMENT_METHOD_EXPIRED",
    "message": "등록된 결제 수단이 만료되었습니다.",
    "retryable": false,
    "requestId": "req_9f21"
  }
}
```

`402`, `409`, `422` 같은 상태 코드는 클라이언트가 대략의 성격을 이해하게 하고, `PAYMENT_METHOD_EXPIRED` 같은 제품 코드는 UI와 운영 정책을 구체화합니다.

## 재시도 가능성을 명시한다

클라이언트가 가장 알고 싶은 정보는 "다시 시도하면 되는가"입니다. 네트워크 문제, 일시적인 외부 API 장애, 락 경합은 재시도할 수 있습니다. 반면 권한 부족, 잘못된 입력, 만료된 토큰은 같은 요청을 반복해도 해결되지 않습니다.

에러 문서에는 코드별 재시도 정책을 포함해야 합니다.

| 코드 | 재시도 | 클라이언트 행동 |
| --- | --- | --- |
| `RATE_LIMITED` | 가능 | `Retry-After` 이후 재시도 |
| `VALIDATION_FAILED` | 불가 | 필드 오류 표시 |
| `UPSTREAM_TIMEOUT` | 가능 | 지수 백오프 |
| `PERMISSION_DENIED` | 불가 | 권한 요청 또는 로그인 안내 |

재시도 가능한 에러라도 무한 재시도는 위험합니다. 최대 횟수, 백오프, 사용자 취소 경로가 함께 필요합니다.

## 사용자 메시지와 로그 메시지를 나눈다

사용자에게 보여줄 문장과 개발자가 볼 상세 원인은 다릅니다. 사용자 메시지에는 민감한 내부 정보를 넣지 않아야 합니다. SQL 오류, 외부 서비스 응답 원문, 스택 트레이스는 로그에만 남겨야 합니다.

대신 응답에는 `requestId`를 포함합니다. 사용자 문의가 들어오면 지원팀은 이 값을 받아 로그에서 상세 원인을 찾을 수 있습니다.

```text
사용자 메시지: 파일을 업로드하지 못했습니다. 잠시 후 다시 시도해 주세요.
로그 메시지: S3 PutObject timeout after 3 retries, bucket=...
```

이 분리는 보안과 지원 효율을 동시에 높입니다.

## 검증 에러는 필드 단위로 반환한다

폼이나 설정 API에서는 어떤 필드가 왜 실패했는지가 중요합니다. 하나의 메시지 문자열로 합치면 클라이언트가 필드별 UI를 만들기 어렵습니다.

```json
{
  "error": {
    "code": "VALIDATION_FAILED",
    "fields": [
      { "path": "email", "code": "INVALID_FORMAT" },
      { "path": "password", "code": "TOO_SHORT", "min": 12 }
    ]
  }
}
```

필드 에러의 `code`는 번역과 접근성에도 도움이 됩니다. 화면 문구는 클라이언트에서 로케일에 맞게 조립할 수 있습니다.

## 에러 코드를 관측 지표로 쓴다

에러 코드는 로그에만 있으면 반쪽입니다. 대시보드와 알림에서도 같은 분류를 써야 합니다. `UPSTREAM_TIMEOUT`이 갑자기 늘면 외부 의존성 문제를 의심할 수 있고, `VALIDATION_FAILED`가 특정 배포 이후 늘면 UI 변경이나 API 계약 변경을 확인할 수 있습니다.

단, 사용자 ID나 입력값처럼 고유값이 많은 데이터를 에러 코드에 넣으면 메트릭 카디널리티가 폭발합니다. 코드는 안정적인 enum처럼 관리해야 합니다.

## 체크리스트

- HTTP 상태 코드와 제품 에러 코드를 분리한다.
- 에러별 재시도 가능성과 클라이언트 행동을 문서화한다.
- 사용자 메시지와 내부 로그 메시지를 분리한다.
- 검증 실패는 필드 단위 구조로 반환한다.
- `requestId`를 응답과 로그에 함께 남긴다.
- 에러 코드를 메트릭 차원으로 사용하되 고유값은 넣지 않는다.

잘 설계된 에러 체계는 장애 대응 시간을 줄이고 클라이언트 구현을 단순하게 만듭니다. 실패를 잘 설명하는 API가 결국 더 안정적인 API입니다.
