로컬 개발 환경이 느린 팀은 기능 개발보다 상태 만들기에 시간을 많이 씁니다. 특정 권한의 사용자, 결제 실패 주문, 만료 직전 구독, 에러가 난 배치 작업을 매번 수동으로 만들면 개발자는 흐름을 잃습니다. 좋은 fixture는 이 비용을 줄입니다.

Fixture는 테스트만을 위한 데이터가 아닙니다. 개발자, QA, 디자이너, 지원팀이 같은 상태를 빠르게 재현하게 만드는 제품 운영 도구입니다.

## 사용자 시나리오 중심으로 만든다

테이블 구조를 그대로 따라가는 seed 데이터는 만들기 쉽지만 쓰기 어렵습니다. 실제로 필요한 것은 "어떤 상황을 재현하는가"입니다.

```text
admin-user: 모든 관리 권한을 가진 사용자
trial-expiring-today: 오늘 체험판이 끝나는 조직
invoice-payment-failed: 결제 실패 상태의 청구서
large-project: 카드 1,000개가 있는 프로젝트
locked-account: 로그인 잠금 상태의 계정
```

이름만 봐도 어떤 화면과 테스트에 쓸지 알 수 있어야 합니다.

## fixture 생성은 멱등적이어야 한다

로컬 데이터를 여러 번 초기화해도 같은 결과가 나와야 합니다. 이미 존재하는 사용자를 또 만들다가 실패하거나, 실행할 때마다 ID가 바뀌면 자동화가 어려워집니다.

가능하면 자연 키나 고정 ID를 사용합니다.

```text
email: admin@example.test
organization_slug: demo-large-project
project_key: fixture-kanban-1000
```

고정된 값은 문서와 테스트에서 참조하기 쉽습니다. 단, 운영 환경과 섞이지 않도록 `.test` 도메인이나 명확한 prefix를 사용해야 합니다.

## 작은 기본 세트와 큰 부하 세트를 나눈다

모든 개발자가 항상 대용량 데이터를 필요로 하지는 않습니다. 기본 fixture는 빠르게 생성되어야 하고, 성능 테스트용 대용량 fixture는 별도 명령으로 분리하는 편이 좋습니다.

```text
npm run db:seed          # 기본 개발 데이터
npm run db:seed:billing  # 결제 시나리오
npm run db:seed:large    # 성능 확인용 대용량 데이터
```

기본 명령이 느려지면 개발자는 seed를 건너뛰기 시작합니다. 그러면 재현 가능한 환경이라는 장점이 사라집니다.

## 외부 서비스 상태도 포함한다

현대 애플리케이션의 상태는 DB에만 있지 않습니다. 이메일 발송, 결제 provider, 검색 인덱스, 오브젝트 스토리지, feature flag도 화면에 영향을 줍니다.

로컬 fixture는 가능한 한 외부 의존성의 대체 상태까지 준비해야 합니다.

- 이메일은 capture inbox에 쌓이게 한다.
- 결제는 sandbox customer와 invoice를 만든다.
- 검색은 fixture 데이터로 인덱스를 재생성한다.
- feature flag는 시나리오별로 고정한다.

DB만 초기화하고 나머지를 수동으로 맞추게 하면 재현성이 떨어집니다.

## 문서보다 명령을 먼저 제공한다

긴 문서에 "이 버튼을 누르고 저 값을 입력하세요"라고 쓰는 것보다 명령 하나가 낫습니다.

```text
npm run scenario:payment-failed
```

명령 실행 후에는 접속 URL, 로그인 계정, 확인할 화면을 출력하면 좋습니다. 개발자는 복잡한 준비 과정을 기억하지 않아도 됩니다.

## 체크리스트

- 테이블이 아니라 사용자 시나리오 기준으로 fixture를 설계한다.
- 여러 번 실행해도 같은 결과가 나오게 만든다.
- 기본 seed와 대용량 seed를 분리한다.
- 외부 서비스 mock, sandbox, index 상태도 함께 준비한다.
- 시나리오별 실행 명령과 접속 정보를 제공한다.
- fixture 계정과 데이터는 운영 데이터와 명확히 구분한다.

좋은 fixture는 팀의 공통 언어입니다. "그 결제 실패 케이스"를 누구나 같은 명령으로 열 수 있을 때 개발과 리뷰 속도가 눈에 띄게 빨라집니다.
