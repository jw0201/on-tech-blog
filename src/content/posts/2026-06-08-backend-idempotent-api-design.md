운영 환경의 API는 한 번만 호출된다는 보장이 없습니다. 클라이언트 타임아웃, 모바일 네트워크 끊김, 프록시 재시도, 작업 큐 재처리 때문에 같은 요청이 두 번 이상 들어올 수 있습니다. 이때 중복 주문, 중복 결제, 중복 작업 생성이 생기지 않게 만드는 핵심이 멱등성입니다.

멱등성은 같은 요청을 여러 번 실행해도 결과가 한 번 실행한 것과 같게 만드는 성질입니다. 조회 API에서는 자연스럽지만, 생성 API에서는 의도적으로 설계해야 합니다.

## 위험한 API부터 찾는다

모든 API에 같은 수준의 멱등성 장치를 붙일 필요는 없습니다. 먼저 중복 실행의 피해가 큰 요청을 찾습니다.

- 결제 승인, 환불, 포인트 지급
- 주문 생성, 예약 확정, 티켓 발급
- 이메일 발송, 푸시 발송, 외부 웹훅 호출
- 장시간 걸리는 배치 작업 생성
- 재고 차감, 쿠폰 사용 처리

이런 API는 "클라이언트가 재시도하지 않으면 된다"로 해결되지 않습니다. 실패했는지 성공했는지 모르는 상태가 반드시 생기기 때문입니다.

## Idempotency-Key를 받는다

생성 요청에는 클라이언트가 만든 멱등성 키를 받는 방식이 실용적입니다.

```http
POST /orders
Idempotency-Key: 6f8d7f86-7f3b-4d62-9b60-5f1a0c9a4b2d
Content-Type: application/json
```

서버는 사용자 ID, API 경로, 멱등성 키를 조합해 요청 처리 결과를 저장합니다. 같은 조합의 요청이 다시 오면 작업을 다시 실행하지 않고 이전 결과를 반환합니다.

키의 범위는 중요합니다. 전역으로만 보면 다른 사용자가 같은 키를 만들었을 때 충돌할 수 있고, 경로를 무시하면 서로 다른 API가 같은 키를 공유할 수 있습니다. 보통은 `user_id + endpoint + idempotency_key` 형태가 안전합니다.

## 요청 본문 해시를 함께 저장한다

같은 멱등성 키로 다른 요청 본문이 들어오면 어떻게 해야 할까요. 이 경우는 재시도가 아니라 클라이언트 버그일 가능성이 큽니다. 서버는 최초 요청의 본문 해시를 저장해 두고, 이후 같은 키로 다른 본문이 오면 `409 Conflict`를 반환하는 편이 좋습니다.

```text
idempotency_key
request_hash
status
response_body
created_at
expires_at
```

이렇게 하면 같은 키가 실수로 재사용되어 잘못된 결과를 돌려주는 일을 막을 수 있습니다.

## 처리 중 상태를 명확히 둔다

중복 요청이 동시에 들어올 수도 있습니다. 첫 요청이 아직 처리 중인데 두 번째 요청이 들어오면, 둘 다 실제 작업을 수행하지 않게 잠금이나 유니크 제약을 사용해야 합니다.

실무에서는 멱등성 테이블에 유니크 인덱스를 걸고 먼저 `processing` 레코드를 삽입합니다. 삽입에 성공한 요청만 실제 작업을 진행하고, 실패한 요청은 기존 레코드 상태를 봅니다.

- `processing`: 아직 처리 중이므로 409 또는 202 반환
- `succeeded`: 저장된 응답 반환
- `failed`: 실패 정책에 따라 저장된 에러 반환 또는 재시도 허용

실패 상태는 신중해야 합니다. 외부 결제사 호출은 성공했지만 내부 DB 저장이 실패한 경우처럼 애매한 상태가 생길 수 있습니다. 이런 API는 reconciliation 작업까지 같이 설계해야 합니다.

## DB 제약은 마지막 방어선이다

멱등성 키만 믿으면 부족합니다. 도메인에도 중복을 막는 제약이 있어야 합니다.

예를 들어 한 장바구니에서 주문은 한 번만 생성되어야 한다면 `cart_id`에 유니크 제약을 둘 수 있습니다. 쿠폰 사용은 `user_id + coupon_id` 유니크 제약이 자연스럽습니다. 멱등성 키는 요청 재시도를 다루고, 도메인 유니크 제약은 비즈니스 중복을 막습니다.

## 만료 정책을 둔다

멱등성 레코드는 영원히 보관할 필요가 없습니다. 결제나 주문처럼 중요한 요청은 며칠에서 몇 주까지 보관할 수 있고, 일반 작업 생성은 24시간 정도로 충분할 수 있습니다.

보관 기간은 클라이언트 재시도 가능 시간, 외부 시스템 정산 주기, 감사 요구사항을 기준으로 정합니다. 만료된 키가 다시 들어오면 새 요청으로 처리한다는 점도 API 문서에 적어야 합니다.

## 체크리스트

- 중복 실행 피해가 큰 API부터 멱등성 키를 적용한다.
- `user_id + endpoint + key` 범위로 중복을 판단한다.
- 요청 본문 해시를 저장해 키 오용을 막는다.
- 처리 중, 성공, 실패 상태를 분리한다.
- 도메인 유니크 제약을 마지막 방어선으로 둔다.
- 멱등성 레코드 만료 정책을 문서화한다.

멱등성은 화려한 기능이 아니라 장애 상황에서 사용자의 돈과 시간을 지키는 기본 설계입니다. 재시도 가능한 시스템을 만들려면 API는 중복 요청을 정상 입력으로 받아들여야 합니다.
