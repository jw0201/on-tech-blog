팀의 개발 생산성은 거창한 플랫폼보다 작은 반복 작업에서 많이 새어 나갑니다. 새 동료가 환경을 맞추는 데 하루를 쓰고, 테스트 명령이 사람마다 다르고, 로컬 DB 초기화 절차가 문서와 실제가 다르면 개발 속도는 꾸준히 느려집니다.

좋은 로컬 스크립트는 팀의 암묵지를 실행 가능한 인터페이스로 바꿉니다.

## 자주 쓰는 작업부터 표준화한다

처음부터 완벽한 CLI를 만들 필요는 없습니다. 반복 횟수가 많고 실패 비용이 큰 작업부터 스크립트로 만듭니다.

```text
npm run dev
npm run test
npm run lint
npm run typecheck
npm run db:reset
npm run verify
```

중요한 것은 이름의 일관성입니다. 저장소마다 테스트 명령이 다르면 자동화와 온보딩이 어려워집니다. 여러 프로젝트를 운영한다면 가능한 한 같은 명령 이름을 유지하는 편이 좋습니다.

## verify 하나로 리뷰 전 확인을 묶는다

Pull request를 올리기 전 사람이 실행해야 하는 검사를 하나의 명령으로 묶으면 실수가 줄어듭니다.

```json
{
  "scripts": {
    "verify": "npm run lint && npm run typecheck && npm run test"
  }
}
```

CI와 로컬의 `verify`가 최대한 같으면 "로컬에서는 됐는데 CI에서 실패"하는 일이 줄어듭니다. 완전히 같게 만들 수 없다면 차이를 문서와 출력에 명확히 남깁니다.

## 실패 메시지를 다음 행동으로 연결한다

스크립트는 실패할 수 있습니다. 이때 단순히 stack trace만 보여주면 새 팀원은 막힙니다. 가능한 경우 실패 원인과 다음 명령을 안내합니다.

```text
Local database is not running.
Run: docker compose up -d postgres
```

이런 작은 메시지가 문서를 뒤지는 시간을 줄입니다. 특히 포트 충돌, 누락된 환경 변수, 오래된 의존성은 친절한 실패 메시지의 효과가 큽니다.

## 스크립트도 유지보수 대상이다

한 번 만든 스크립트를 방치하면 더 위험합니다. 실제로는 쓰이지 않는 명령, 깨진 옵션, 오래된 문서 링크가 남아 있으면 팀은 다시 개인별 절차로 돌아갑니다.

분기별로 한 번쯤 새 checkout에서 `setup`, `dev`, `verify`가 동작하는지 확인하는 것이 좋습니다. 온보딩 중 막힌 지점은 스크립트 개선의 가장 좋은 입력입니다.

## 체크리스트

- 반복적이고 실패 비용이 큰 작업부터 스크립트로 만든다.
- 저장소 간 명령 이름을 가능하면 통일한다.
- 리뷰 전 검사는 `verify` 같은 단일 명령으로 묶는다.
- 실패 메시지는 다음 행동을 알려준다.
- 온보딩에서 막힌 절차를 스크립트로 흡수한다.

좋은 개발 스크립트는 팀원이 외워야 할 절차를 줄입니다. 기억해야 할 것이 줄어들수록 개발자는 제품과 코드에 더 많은 집중력을 쓸 수 있습니다.
