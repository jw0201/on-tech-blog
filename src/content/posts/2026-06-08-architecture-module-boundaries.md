작은 서비스에서 모듈 경계를 너무 일찍 세우면 코드보다 폴더 구조가 더 복잡해집니다. 반대로 아무 경계 없이 기능을 계속 추가하면 어느 순간 작은 변경도 여러 파일을 뒤흔듭니다. 좋은 경계는 처음부터 거창한 아키텍처를 적용하는 것이 아니라, 변경 이유가 다른 코드를 자연스럽게 떨어뜨리는 일입니다.

## 경계는 명사보다 변경 이유로 잡는다

폴더 이름을 도메인 명사로 나누는 것만으로는 부족합니다. `user`, `order`, `payment`처럼 나눴더라도 실제 변경이 항상 같이 일어난다면 경계가 약한 것입니다.

먼저 이런 질문을 해 봅니다.

- 이 코드는 어떤 비즈니스 규칙이 바뀔 때 수정되는가
- 어떤 팀이나 담당자가 주로 바꾸는가
- 어떤 외부 시스템 장애에 영향을 받는가
- 테스트할 때 어떤 의존성을 mock해야 하는가

예를 들어 결제 승인 로직과 결제 내역 화면 조회 로직은 둘 다 payment에 있지만 변경 이유가 다릅니다. 승인 로직은 외부 결제사, 재시도, 정산 규칙의 영향을 받고, 조회 로직은 화면 요구사항과 필터 조건의 영향을 받습니다.

## public API를 작게 만든다

모듈 경계를 세운다는 것은 폴더를 나누는 것보다 "밖에서 무엇을 호출할 수 있는가"를 정하는 일에 가깝습니다.

```text
payments/
  index.ts
  approve-payment.ts
  refund-payment.ts
  payment-repository.ts
  provider-client.ts
```

외부에서는 `payments/index.ts`가 export하는 함수만 쓰게 하고, 내부 파일을 직접 import하지 않게 합니다. 이렇게 하면 내부 구현을 바꿔도 호출자 영향이 작습니다.

```ts
export { approvePayment } from "./approve-payment";
export { refundPayment } from "./refund-payment";
```

규칙이 강하게 필요해지면 lint rule이나 path alias로 내부 import를 막을 수 있습니다. 처음에는 code review 규칙만으로도 충분합니다.

## DB 모델을 그대로 경계로 쓰지 않는다

테이블이 있다고 모듈이 되는 것은 아닙니다. `users` 테이블을 여러 기능이 쓴다면 user 모듈이 모든 것을 소유하게 만들기보다, 각 기능이 필요한 읽기 모델을 따로 갖는 편이 나을 때가 많습니다.

예를 들어 알림 발송은 사용자 전체 정보가 아니라 `user_id`, `email`, `notification_opt_in`만 필요합니다. 이 경우 알림 모듈이 사용자 모듈의 거대한 entity를 직접 다루게 하기보다 필요한 조회 함수를 통해 작은 모델만 받게 하면 결합이 줄어듭니다.

## 외부 시스템은 어댑터 뒤에 둔다

결제사, 이메일 발송, object storage, AI API처럼 외부 시스템에 붙는 코드는 경계가 특히 중요합니다. 외부 SDK 타입이 서비스 내부 깊숙이 퍼지면 공급자를 바꾸거나 테스트하기 어려워집니다.

```ts
type SendEmailInput = {
  to: string;
  subject: string;
  html: string;
};

export interface EmailSender {
  send(input: SendEmailInput): Promise<void>;
}
```

서비스 코드는 `EmailSender`에 의존하고, 실제 SDK 호출은 adapter가 맡습니다. 작은 프로젝트에서도 이 정도 추상화는 테스트 비용을 줄여 줍니다.

## 순환 참조를 경고로 본다

모듈 A가 B를 import하고, B도 A를 import하기 시작하면 경계가 흐려졌다는 신호입니다. 순환 참조 자체가 항상 즉시 장애를 만들지는 않지만, 변경 방향이 꼬였다는 뜻일 때가 많습니다.

해결 방법은 보통 세 가지입니다.

- 공통 타입이나 helper를 더 낮은 레이어로 내린다.
- 한쪽 호출 방향을 이벤트나 callback으로 바꾼다.
- 실제로 같은 모듈이어야 하는지 다시 본다.

순환 참조를 억지로 우회하기보다 왜 서로 알아야 하는지 보는 것이 중요합니다.

## 체크리스트

- 모듈은 명사보다 변경 이유와 소유권으로 나눈다.
- 외부 호출자는 `index.ts` 같은 작은 public API만 보게 한다.
- DB 테이블 구조를 그대로 모듈 경계로 삼지 않는다.
- 외부 SDK 타입은 adapter 밖으로 새지 않게 한다.
- 순환 참조가 생기면 경계가 맞는지 다시 검토한다.
- 경계 강제는 처음부터 과하게 하지 말고 필요할 때 lint로 올린다.

작은 서비스의 좋은 아키텍처는 복잡한 패턴 이름에서 나오지 않습니다. 같이 바뀌는 코드는 가깝게, 다르게 바뀌는 코드는 멀게 두는 감각에서 시작합니다.
