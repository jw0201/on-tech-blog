웹 성능 문제의 상당 부분은 이미지에서 시작됩니다. 큰 hero 이미지, 원본 크기 썸네일, 불필요한 PNG, 잘못된 lazy loading이 LCP와 데이터 사용량을 악화시킵니다. 이미지는 눈에 보이는 자산이라 줄이기 어렵지만, 예산 없이 넣기 시작하면 페이지는 금방 무거워집니다.

이미지 성능 예산은 "최대한 줄이자"가 아니라 "어떤 화면에서 얼마까지 쓸 수 있는가"를 정하는 일입니다.

## 화면 단위로 예산을 둔다

사이트 전체에 이미지 총량 하나만 정하면 실무에서 쓰기 어렵습니다. 홈, 상세, 목록, 대시보드처럼 화면 성격에 따라 필요한 이미지가 다릅니다.

예산은 다음처럼 화면 단위로 잡는 편이 좋습니다.

```text
홈 첫 화면:
- LCP 이미지: 180KB 이하
- 첫 화면 전체 이미지: 350KB 이하
- below-the-fold 이미지: lazy loading

상품 목록:
- 썸네일 1개당 40KB 이하
- 첫 페이지 이미지 총량 500KB 이하
```

숫자는 서비스의 사용자 환경에 맞춰 조정해야 합니다. 모바일 사용자가 많거나 해외 네트워크 비중이 높다면 더 보수적으로 잡아야 합니다.

## LCP 이미지는 특별 취급한다

첫 화면의 가장 큰 이미지는 LCP에 직접 영향을 줍니다. 이 이미지는 lazy loading 대상이 아닙니다. 브라우저가 늦게 발견하지 않도록 preload나 priority 힌트를 검토해야 합니다.

```html
<img
  src="/images/hero.avif"
  width="1200"
  height="675"
  fetchpriority="high"
  alt="제품 대시보드 화면"
/>
```

반대로 화면 아래의 이미지는 lazy loading을 적용합니다. 모든 이미지에 같은 로딩 전략을 붙이는 것보다, LCP 후보와 나머지를 구분하는 것이 중요합니다.

## 실제 표시 크기에 맞춘다

가로 300px로 보이는 썸네일에 2400px 원본을 내려보내는 문제는 흔합니다. 반응형 이미지는 `srcset`과 `sizes`로 브라우저가 적절한 크기를 고르게 해야 합니다.

```html
<img
  src="/thumb-640.webp"
  srcset="/thumb-320.webp 320w, /thumb-640.webp 640w, /thumb-960.webp 960w"
  sizes="(max-width: 640px) 45vw, 240px"
  alt="프로젝트 썸네일"
/>
```

이미지 CDN이나 빌드 파이프라인이 있다면 width, quality, format 변환을 자동화합니다. 사람이 매번 수동으로 리사이즈하면 오래 유지하기 어렵습니다.

## 포맷은 콘텐츠에 맞춘다

사진에는 AVIF나 WebP가 대체로 유리합니다. 투명도가 있는 UI 그래픽도 WebP가 잘 맞을 수 있습니다. 단순 아이콘은 SVG가 좋고, 복잡한 스크린샷은 포맷별 품질을 비교해야 합니다.

PNG는 무조건 나쁜 포맷이 아니지만, 사진이나 대형 스크린샷을 PNG로 배포하는 것은 비용이 큽니다. 이미지 예산을 지키려면 업로드 단계에서 포맷 변환을 기본값으로 두는 편이 좋습니다.

## 회귀를 CI에서 잡는다

이미지 최적화는 한 번 해도 새 이미지가 추가되면 다시 나빠집니다. PR 단계에서 이미지 크기와 포맷을 검사하면 회귀를 빨리 잡을 수 있습니다.

- 특정 디렉터리의 원본 이미지 최대 크기 제한
- PNG 사용 경고
- width와 height 속성 누락 검사
- Lighthouse 또는 WebPageTest 기반 LCP 회귀 확인
- 번들 리포트에 이미지 총량 표시

자동 검사가 있으면 리뷰어가 매번 눈으로 이미지 용량을 확인하지 않아도 됩니다.

## 체크리스트

- 화면별 이미지 예산을 숫자로 정한다.
- LCP 이미지는 lazy loading하지 않고 우선순위를 높인다.
- `srcset`과 `sizes`로 실제 표시 크기에 맞춘 이미지를 제공한다.
- 콘텐츠에 맞는 포맷을 선택하고 변환을 자동화한다.
- width와 height를 명시해 레이아웃 shift를 줄인다.
- CI에서 이미지 크기와 포맷 회귀를 검사한다.

이미지 최적화는 화질을 포기하는 일이 아닙니다. 사용자가 보는 품질은 지키면서 필요 없는 바이트를 보내지 않도록 설계하는 일입니다.
