비동기 작업 큐는 API 응답 시간을 줄이고 외부 시스템 장애를 흡수하는 데 유용합니다. 하지만 큐에 넣었다는 사실만으로 안정성이 생기지는 않습니다. 재시도 정책, 실패 보관, 멱등성, 모니터링이 없으면 큐는 조용히 실패를 쌓는 장소가 됩니다.

이메일 발송, 이미지 변환, 웹훅 전송, 리포트 생성처럼 비동기 작업을 운영할 때는 실패를 정상 시나리오로 보고 설계해야 합니다.

## 작업은 작고 명확하게 만든다

큐 작업 payload에는 실행에 필요한 최소 정보만 담는 것이 좋습니다.

```json
{
  "job_id": "job_123",
  "type": "send_welcome_email",
  "user_id": "user_456"
}
```

사용자 이메일, 이름, 플랜 정보 전체를 payload에 복사하면 작업 실행 시점의 최신 상태와 어긋날 수 있고, 민감 정보가 큐에 오래 남을 수 있습니다. 보통은 식별자만 넣고 worker가 실행 시점에 필요한 데이터를 다시 읽습니다.

단, 외부로 반드시 당시 값 그대로 보내야 하는 작업이라면 snapshot을 명시적으로 저장합니다. 중요한 것은 어떤 값이 참조이고 어떤 값이 스냅샷인지 의도적으로 정하는 것입니다.

## 재시도 가능한 실패와 아닌 실패를 나눈다

모든 실패를 같은 방식으로 재시도하면 문제가 커집니다.

재시도할 만한 실패:

- 외부 API timeout
- 429 rate limit
- 일시적인 네트워크 오류
- DB deadlock
- object storage 일시 장애

재시도하지 않을 실패:

- 잘못된 입력
- 권한 없음
- 존재하지 않는 리소스
- template 누락
- 영구적으로 유효하지 않은 이메일 주소

worker는 에러 종류를 구분해 재시도 여부를 결정해야 합니다. 무조건 10번 재시도하면 처리량만 잡아먹고 실제 문제 해결은 늦어집니다.

## exponential backoff와 jitter를 쓴다

외부 서비스가 잠시 느려졌을 때 모든 작업을 즉시 재시도하면 장애를 더 키울 수 있습니다. 재시도 간격은 점점 늘리고, jitter를 넣어 재시도 시점을 흩뜨리는 것이 좋습니다.

```text
1차 실패: 10초 후
2차 실패: 1분 후
3차 실패: 5분 후
4차 실패: 30분 후
```

작업 유형마다 최대 재시도 횟수도 달라야 합니다. 이메일 발송은 몇 번 늦어져도 괜찮지만, 사용자 요청 직후 필요한 작업은 짧게 실패시키고 사용자에게 다시 시도하게 하는 편이 나을 수 있습니다.

## 멱등성을 작업 안에 넣는다

큐는 at-least-once delivery인 경우가 많습니다. 즉 같은 작업이 두 번 실행될 수 있습니다. worker는 중복 실행되어도 결과가 안전해야 합니다.

예를 들어 포인트 지급 작업이라면 `user_id + reason + reference_id`에 유니크 제약을 둡니다. 웹훅 전송은 delivery id를 기록하고 같은 delivery가 다시 실행될 때 중복 전송을 막거나, 수신자가 멱등 처리할 수 있는 id를 함께 보냅니다.

```text
job_id는 실행 추적용이고,
business idempotency key는 도메인 중복 방지용이다.
```

이 둘을 혼동하지 않는 것이 중요합니다.

## DLQ를 방치하지 않는다

최대 재시도 후에도 실패한 작업은 dead letter queue나 failed jobs 테이블로 이동시킵니다. 하지만 DLQ는 쓰레기통이 아니라 운영 대기열입니다.

DLQ에는 최소한 다음 정보가 있어야 합니다.

- 작업 유형
- payload
- 마지막 에러 코드와 메시지
- 실패 횟수
- 최초 생성 시각
- 마지막 시도 시각
- correlation id

그리고 DLQ에 작업이 쌓이면 알림이 가야 합니다. 아무도 보지 않는 DLQ는 실패를 숨기는 장치일 뿐입니다.

## 관측 지표를 만든다

큐 운영에서 봐야 할 지표는 단순히 worker가 살아 있는지가 아닙니다.

- queue depth
- oldest job age
- success rate
- retry rate
- dead letter count
- job duration p95
- worker concurrency

특히 oldest job age는 중요합니다. queue depth가 작아도 오래된 작업이 남아 있다면 특정 작업 유형이 계속 실패하고 있을 수 있습니다.

## 체크리스트

- payload에는 최소 식별자만 담고 snapshot 여부를 명확히 한다.
- 재시도 가능한 실패와 영구 실패를 구분한다.
- backoff와 jitter로 재시도 폭주를 막는다.
- worker는 중복 실행되어도 안전하게 만든다.
- DLQ에는 재처리에 필요한 정보를 남긴다.
- DLQ 적체와 oldest job age에 알림을 건다.
- 작업 유형별 재시도 횟수와 시간 제한을 다르게 둔다.

비동기 큐는 실패를 없애는 도구가 아니라 실패를 다룰 시간을 벌어 주는 도구입니다. 그 시간을 제대로 쓰려면 재시도, 멱등성, 관측성이 처음부터 함께 설계되어야 합니다.
