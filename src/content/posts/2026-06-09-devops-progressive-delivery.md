배포는 코드가 main branch에 merge되는 순간 끝나지 않습니다. 사용자가 실제로 새 동작을 만나고, 지표가 정상이며, 문제가 생겼을 때 빠르게 되돌릴 수 있어야 배포가 끝난 것입니다. Progressive delivery는 이 과정을 한 번에 전체 사용자에게 열지 않고 작은 범위부터 넓혀 가는 접근입니다.

핵심은 변경을 작게 노출하고, 관측하고, 자동 또는 수동으로 멈출 기준을 정하는 것입니다.

## 배포와 공개를 분리한다

가장 먼저 할 일은 코드 배포와 기능 공개를 분리하는 것입니다. Feature flag를 사용하면 코드는 운영에 배포하되 특정 사용자, 조직, 지역, 비율에만 새 기능을 켤 수 있습니다.

이 분리는 롤백 비용을 줄입니다. 코드 배포 자체를 되돌리지 않고도 기능을 끌 수 있기 때문입니다. 데이터 마이그레이션이나 외부 연동이 섞인 변경일수록 이 차이가 큽니다.

## 카나리는 사용자 영향이 작은 곳부터 시작한다

Canary 배포는 새 버전을 일부 트래픽에만 노출하는 방식입니다. 중요한 것은 비율보다 관측 가능성입니다. 1%만 열어도 그 1%에서 에러율, latency, 주요 비즈니스 이벤트를 볼 수 있어야 합니다.

좋은 시작점은 내부 사용자, 베타 조직, 트래픽이 낮은 지역, 특정 API consumer입니다. 단, 내부 사용자만으로는 실제 부하와 데이터 다양성을 대표하지 못할 수 있으므로 결국 실제 사용자 일부까지 확장해야 합니다.

## 멈출 조건을 미리 정한다

배포 중 문제가 생겼을 때 회의가 시작되면 늦습니다. 배포 전에 멈출 조건을 정해야 합니다.

- 5xx 비율이 기준 대비 2배 이상 증가
- p95 latency가 10분 이상 임계치 초과
- 결제, 가입, 생성 같은 핵심 전환율 급락
- 특정 에러 로그가 분당 N회 이상 발생
- 고객 지원 인입이 짧은 시간에 증가

이 조건은 대시보드와 알림에 연결되어야 합니다. 문서에만 있는 기준은 실제 배포 중에는 잘 쓰이지 않습니다.

## 롤백도 리허설한다

롤백은 버튼 이름을 아는 것과 실제로 성공하는 것이 다릅니다. 스키마 변경, 캐시, 큐 메시지, 모바일 앱 호환성 때문에 단순한 이전 버전 배포가 실패할 수 있습니다.

큰 변경 전에는 다음을 확인합니다.

- 이전 버전이 새 DB 스키마와 함께 동작하는가
- feature flag를 끄면 새 코드 경로가 완전히 비활성화되는가
- 새로 발행한 이벤트를 이전 consumer가 무시할 수 있는가
- 캐시된 새 응답을 이전 UI가 처리할 수 있는가

## 체크리스트

- 코드 배포와 기능 공개를 feature flag로 분리한다.
- 작은 사용자 집단에서 시작해 점진적으로 확장한다.
- 에러율, latency, 핵심 비즈니스 지표를 같이 본다.
- 배포 전 중단 조건과 담당자를 정한다.
- 롤백 경로를 실제 제약과 함께 검증한다.

Progressive delivery는 배포를 느리게 만드는 절차가 아닙니다. 문제가 생겼을 때 영향 범위를 작게 유지하기 때문에 오히려 팀이 더 자주, 더 자신 있게 배포할 수 있게 해줍니다.
