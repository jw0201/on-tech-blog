세션 쿠키는 웹 애플리케이션 인증의 핵심입니다. 사용자가 로그인했다는 사실을 브라우저와 서버가 공유하는 작은 값이지만, 탈취되면 계정 접근 권한이 그대로 넘어갈 수 있습니다. 그래서 세션 쿠키는 발급 로직만큼 속성 설정이 중요합니다.

쿠키 보안은 특별한 기술보다 기본값을 올바르게 잡는 일에 가깝습니다. `HttpOnly`, `Secure`, `SameSite`, 만료 정책만 제대로 적용해도 흔한 공격 표면을 크게 줄일 수 있습니다.

## HttpOnly로 스크립트 접근을 막는다

`HttpOnly`가 붙은 쿠키는 브라우저의 JavaScript에서 읽을 수 없습니다. XSS 취약점이 있더라도 공격자가 `document.cookie`로 세션 값을 바로 가져가기 어렵게 만듭니다.

```http
Set-Cookie: session=abc123; HttpOnly
```

물론 `HttpOnly`가 XSS 자체를 막지는 않습니다. 공격자는 피해자의 브라우저에서 요청을 보낼 수 있습니다. 그래도 세션 값의 직접 탈취를 줄이는 중요한 방어층입니다.

## Secure로 HTTPS에서만 전송한다

`Secure` 속성은 쿠키가 HTTPS 연결에서만 전송되게 합니다. 운영 환경의 인증 쿠키에는 반드시 필요합니다.

```http
Set-Cookie: session=abc123; HttpOnly; Secure
```

로컬 개발에서 HTTP를 사용한다면 환경별 설정이 필요할 수 있습니다. 하지만 운영 환경에서 `Secure`가 빠지는 것은 배포 차단 조건으로 두는 편이 좋습니다.

## SameSite로 CSRF 위험을 낮춘다

`SameSite`는 다른 사이트에서 시작된 요청에 쿠키를 보낼지 결정합니다. 일반적인 웹 앱의 세션 쿠키는 `Lax`가 좋은 기본값입니다.

```http
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax
```

`Strict`는 더 강하지만 외부 링크에서 들어온 직후 로그인 상태가 기대와 다르게 보일 수 있습니다. `None`은 크로스 사이트 쿠키가 필요한 경우에만 쓰고, 반드시 `Secure`와 함께 사용해야 합니다.

CSRF 방어가 SameSite 하나로 끝나는 것은 아닙니다. 상태 변경 요청에는 CSRF 토큰, Origin 검증, 재인증 같은 방어를 상황에 맞게 추가해야 합니다.

## 만료 시간을 의도적으로 정한다

세션 쿠키의 수명은 보안과 사용자 경험의 균형입니다. 너무 짧으면 사용자가 자주 로그아웃되고, 너무 길면 탈취된 세션의 피해 시간이 늘어납니다.

보통은 두 종류의 만료를 함께 둡니다.

- idle timeout: 사용이 없으면 만료
- absolute timeout: 활동과 무관하게 최대 수명 만료

예를 들어 업무용 관리자 페이지는 30분 idle timeout과 12시간 absolute timeout을 둘 수 있습니다. 일반 소비자 서비스는 더 긴 수명이 필요할 수 있지만, 민감 작업에는 재인증을 요구하는 편이 좋습니다.

## 세션 회전을 구현한다

로그인 직후, 권한 상승 직후, 비밀번호 변경 직후에는 세션 ID를 새로 발급해야 합니다. 이를 세션 회전이라고 합니다. 공격자가 로그인 전 세션을 고정해두는 session fixation 공격을 줄일 수 있습니다.

또한 로그아웃과 비밀번호 변경 시 기존 세션을 서버에서 무효화해야 합니다. 쿠키를 브라우저에서 지우는 것만으로는 충분하지 않습니다. 서버의 세션 저장소나 토큰 버전도 함께 갱신해야 합니다.

## 체크리스트

- 인증 쿠키에 `HttpOnly`를 적용한다.
- 운영 환경에서는 `Secure`가 빠진 배포를 차단한다.
- 기본 `SameSite`는 `Lax`로 두고 예외를 명확히 관리한다.
- idle timeout과 absolute timeout을 함께 설계한다.
- 로그인, 권한 변경, 비밀번호 변경 시 세션을 회전한다.
- 로그아웃과 보안 이벤트에서 서버 세션을 무효화한다.

세션 쿠키 보안은 한 번 설정하고 잊는 항목이 아닙니다. 인증 흐름이 바뀔 때마다 쿠키 속성과 만료 정책도 함께 리뷰해야 합니다.
