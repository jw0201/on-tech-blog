서비스가 하나일 때는 통합 테스트만으로도 많은 문제를 잡을 수 있습니다. 하지만 서비스가 나뉘면 장애의 모양이 달라집니다. 제공자 서비스는 정상 배포됐는데 소비자 서비스가 기대하던 필드가 사라져 화면이 깨지는 식입니다. 계약 테스트는 이런 경계 문제를 줄이기 위한 테스트입니다.

계약 테스트의 목표는 "전체 시스템을 매번 다 띄우기"가 아니라 "서비스 사이의 약속이 깨졌는지 빠르게 확인하기"입니다.

## 계약은 소비자의 기대에서 시작한다

API 문서는 제공자가 쓰지만, 실제 장애는 소비자의 기대가 깨질 때 발생합니다. 계약 테스트는 소비자가 어떤 요청을 보내고 어떤 응답을 필요로 하는지 명시하는 방식으로 시작하는 것이 좋습니다.

예를 들어 주문 화면은 사용자 API에서 다음 필드만 필요할 수 있습니다.

```json
{
  "id": "user_123",
  "name": "Jiwon",
  "grade": "pro"
}
```

제공자 API가 내부적으로 더 많은 필드를 반환해도 소비자 계약에는 필요한 필드만 적습니다. 그래야 불필요한 변경에 테스트가 흔들리지 않습니다.

## E2E 테스트와 역할이 다르다

계약 테스트는 E2E 테스트를 대체하지 않습니다. 둘은 잡는 문제가 다릅니다.

- E2E 테스트: 실제 사용자 흐름이 끝까지 동작하는지 확인
- 계약 테스트: 서비스 간 요청과 응답의 약속이 깨졌는지 확인

E2E 테스트는 느리고 실패 원인 파악이 어렵습니다. 계약 테스트는 빠르고 경계가 명확합니다. 중요한 사용자 흐름은 E2E로 남기고, 서비스 간 조합이 많은 부분은 계약 테스트로 보강하는 편이 현실적입니다.

## 제공자 CI에서 계약을 검증한다

계약 테스트의 효과를 보려면 제공자 서비스가 배포되기 전에 소비자 계약을 검증해야 합니다. 제공자 repository의 CI가 소비자 계약 파일을 가져와 현재 API 구현과 비교하는 구조가 좋습니다.

흐름은 단순합니다.

1. 소비자 서비스가 기대하는 요청과 응답 계약을 발행한다.
2. 제공자 서비스 CI가 계약을 가져온다.
3. 제공자 테스트 환경에서 계약 요청을 실행한다.
4. 응답이 계약을 만족하지 않으면 merge나 배포를 막는다.

이렇게 해야 제공자가 "우리 테스트는 통과했다"고 생각한 변경이 소비자를 깨뜨리는 일을 줄일 수 있습니다.

## 계약에는 너무 많은 것을 넣지 않는다

계약 테스트가 실패를 자주 만들면 팀은 곧 무시하게 됩니다. 그래서 계약은 소비자가 정말 의존하는 것만 담아야 합니다.

좋은 계약의 예:

- 필수 필드 존재 여부
- 필드 타입
- 상태 코드
- 에러 코드 형태
- 특정 조건에서의 응답 구조

나쁜 계약의 예:

- 응답 필드 전체 스냅샷
- 정렬이 중요하지 않은 배열의 완전한 순서
- 소비자가 쓰지 않는 내부 필드
- 매번 바뀌는 timestamp의 정확한 값

계약은 API의 모든 세부사항을 얼리는 도구가 아닙니다. 소비자에게 필요한 안정성을 문서화하는 도구입니다.

## 버전 전략과 함께 운영한다

계약 테스트가 있다고 해서 breaking change가 불가능한 것은 아닙니다. 대신 breaking change를 드러내고, 마이그레이션 순서를 정하게 해 줍니다.

안전한 순서는 보통 이렇습니다.

1. 제공자가 새 필드를 추가하고 기존 필드는 유지한다.
2. 소비자가 새 필드로 이동한다.
3. 소비자 계약에서 기존 필드 의존을 제거한다.
4. 제공자가 기존 필드를 삭제한다.

계약 테스트는 이 순서를 CI에서 강제하는 역할을 합니다.

## 체크리스트

- 소비자별로 실제 의존하는 요청과 응답만 계약에 담는다.
- 제공자 CI에서 소비자 계약을 검증한다.
- E2E 테스트와 계약 테스트의 역할을 분리한다.
- 전체 응답 스냅샷보다 필수 필드와 타입을 검증한다.
- breaking change는 추가, 이동, 제거 순서로 진행한다.
- 계약 실패가 배포 전에 보이도록 pipeline에 연결한다.

서비스가 많아질수록 장애는 코드 내부보다 경계에서 자주 생깁니다. 계약 테스트는 그 경계를 느슨하게 방치하지 않고, 자동으로 확인 가능한 약속으로 바꾸는 방법입니다.
