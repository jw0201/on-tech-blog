LLM 기능은 코드만으로 동작이 결정되지 않습니다. 시스템 프롬프트, few-shot 예시, 모델 이름, temperature, 검색 컨텍스트가 모두 결과를 바꿉니다. 그런데 이 요소들을 운영 콘솔에 직접 고쳐 넣기만 하면 나중에 어떤 변경이 품질을 바꿨는지 추적하기 어렵습니다.

AI 기능도 제품 기능입니다. 그래서 프롬프트 변경은 코드 변경처럼 버전 관리되어야 합니다.

## 프롬프트를 파일로 둔다

프롬프트가 코드 안의 긴 문자열이나 관리 콘솔의 텍스트 박스에만 있으면 리뷰하기 어렵습니다. 가능하면 저장소 안에 명시적인 파일로 둡니다.

```text
prompts/
  support-answer.v1.md
  issue-triage.v3.md
  release-note-writer.v2.md
```

파일로 두면 pull request에서 diff를 볼 수 있고, 배포된 버전과 실험 버전을 구분하기 쉽습니다. 프롬프트 안에는 목적, 입력 형식, 출력 형식, 금지 사항, 예외 처리 방식을 명확히 적습니다.

## 모델 설정도 함께 고정한다

프롬프트만 버전 관리하고 모델 설정을 따로 두면 재현성이 떨어집니다. 같은 프롬프트라도 모델, temperature, max token, tool 사용 여부가 달라지면 결과가 달라집니다.

```json
{
  "prompt": "support-answer.v1.md",
  "model": "example-model",
  "temperature": 0.2,
  "maxOutputTokens": 900
}
```

운영에서 실제 사용한 설정을 로그에 남기면 나중에 품질 이슈가 생겼을 때 특정 버전으로 되돌아가기 쉽습니다.

## 작은 평가셋을 만든다

프롬프트 변경은 느낌만으로 배포하기 어렵습니다. 최소한 대표 입력과 기대 기준을 모은 평가셋이 필요합니다.

- 정상 케이스
- 애매한 입력
- 거절해야 하는 입력
- 긴 컨텍스트
- 이전에 실패했던 회귀 케이스

처음부터 거창한 자동 채점 시스템이 필요하지는 않습니다. 입력, 기대 행동, 실제 출력, 사람이 내린 판정만 기록해도 변경 품질을 비교할 수 있습니다.

## 릴리즈 노트를 남긴다

프롬프트 변경 PR에는 "무엇을 바꿨는가"보다 "어떤 실패를 줄이려는가"가 중요합니다.

```text
변경 목적: 환불 정책 질문에서 약관 링크를 누락하는 문제 감소
영향 범위: support-answer
평가 결과: 24개 케이스 중 22개 통과, 기존 대비 링크 누락 4건 감소
위험: 답변 길이가 평균 12% 증가
```

이 정도만 남겨도 몇 주 뒤 품질 변화의 맥락을 되찾을 수 있습니다.

## 체크리스트

- 프롬프트를 저장소 파일로 관리한다.
- 모델명과 generation 설정도 함께 버전 고정한다.
- 대표 입력과 회귀 케이스로 작은 평가셋을 만든다.
- 변경 목적과 평가 결과를 PR에 남긴다.
- 운영 로그에 프롬프트 버전과 모델 설정을 기록한다.

LLM 기능의 품질 개선은 우연한 문장 수정이 아니라 반복 가능한 실험이어야 합니다. 프롬프트를 버전 관리하면 "이번에는 왠지 더 좋아졌다"에서 "어떤 케이스가 왜 좋아졌는지 안다"로 넘어갈 수 있습니다.
