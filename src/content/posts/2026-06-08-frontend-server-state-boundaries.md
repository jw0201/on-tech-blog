프론트엔드 상태 관리가 복잡해지는 순간은 보통 상태 라이브러리를 잘못 골랐을 때가 아니라, 성격이 다른 상태를 한 바구니에 넣었을 때입니다. 서버에서 온 데이터, 사용자가 입력 중인 값, 모달이 열렸는지 같은 화면 상태는 생명주기와 실패 방식이 다릅니다.

## 먼저 상태의 소유자를 나눈다

상태를 만들기 전에 "이 값의 원본은 어디인가"를 먼저 정하면 구조가 단순해집니다.

- 서버 상태: API, DB, 외부 시스템이 원본인 값
- 폼 상태: 사용자가 제출하기 전까지 화면이 잠시 소유하는 값
- UI 상태: 탭, 모달, 토스트, 드롭다운처럼 화면 표현만 위한 값
- 파생 상태: 기존 값으로 계산할 수 있는 값

서버 상태를 전역 store에 복사해 두면 처음에는 편하지만 금방 동기화 문제가 생깁니다. 목록 API를 다시 불렀는데 상세 화면의 store 값은 그대로 남아 있거나, 낙관적 업데이트 실패 후 되돌릴 기준이 애매해집니다.

## 서버 상태는 캐시로 다룬다

서버 상태에는 "가져오기", "캐시", "재검증", "무효화", "로딩", "에러"가 붙습니다. 그래서 직접 `useState`와 전역 store로 관리하기보다 TanStack Query 같은 서버 상태 도구를 쓰는 편이 낫습니다.

예를 들어 사용자 목록을 전역 store에 넣는 대신 query key로 식별합니다.

```tsx
const usersQuery = useQuery({
  queryKey: ["users", filters],
  queryFn: () => fetchUsers(filters),
});
```

중요한 점은 query key에 서버 응답을 결정하는 입력을 모두 넣는 것입니다. 필터, 페이지, 정렬 기준이 key에 빠지면 캐시가 틀린 데이터를 보여줄 수 있습니다.

## 폼 상태는 제출 전까지 고립시킨다

폼은 사용자가 아직 확정하지 않은 값을 담습니다. 따라서 서버 상태와 바로 섞지 않는 편이 좋습니다. 상세 API로 받은 데이터를 폼의 초기값으로 넣을 수는 있지만, 입력 중인 값은 폼 레이어가 따로 관리해야 합니다.

```tsx
const form = useForm({
  defaultValues: {
    name: user.name,
    role: user.role,
  },
});
```

저장 버튼을 누른 뒤 mutation이 성공하면 서버 상태 캐시를 무효화하거나, 응답값으로 캐시를 갱신합니다. 이 흐름을 지키면 "입력 중인 값"과 "서버에 저장된 값"이 헷갈리지 않습니다.

## UI 상태는 가까운 곳에 둔다

모달 열림 여부, 선택된 탭, hover로 표시하는 보조 UI는 대개 컴포넌트 근처에 두는 게 가장 읽기 쉽습니다. 여러 화면이 공유하지 않는다면 전역 store로 올릴 이유가 없습니다.

```tsx
const [isFilterOpen, setFilterOpen] = useState(false);
```

전역으로 올릴 만한 UI 상태는 라우트 전환 후에도 유지되어야 하거나, 앱 전체에서 같은 규칙으로 제어되어야 하는 경우입니다. 예를 들면 현재 로그인 사용자, 테마, 전역 알림 큐 정도입니다.

## 파생 상태는 저장하지 않는다

서버에서 받은 `items`가 있고 화면에 보여줄 `filteredItems`를 만들 수 있다면, `filteredItems`를 별도 상태로 저장하지 않는 편이 안전합니다.

```tsx
const filteredItems = useMemo(
  () => items.filter((item) => item.status === selectedStatus),
  [items, selectedStatus],
);
```

계산 비용이 작다면 `useMemo`도 필요 없습니다. 핵심은 같은 사실을 두 군데 저장하지 않는 것입니다.

## 체크리스트

- API 응답은 서버 상태 도구의 캐시에 둔다.
- 사용자가 제출하기 전 값은 폼 상태로 격리한다.
- 화면 표현만 위한 값은 가능한 한 가까운 컴포넌트에 둔다.
- 기존 값으로 계산 가능한 값은 상태로 저장하지 않는다.
- 전역 store는 앱 전체 정책이나 오래 유지해야 하는 클라이언트 상태에만 쓴다.

상태 관리는 도구 선택보다 경계 설정이 먼저입니다. 상태의 소유자와 수명을 분리하면 화면 코드는 훨씬 덜 흔들립니다.
