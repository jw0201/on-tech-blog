프론트엔드 상태는 모두 같은 성격이 아닙니다. 어떤 상태는 새로고침 후에도 유지되어야 하고, 어떤 상태는 뒤로 가기 버튼과 함께 움직여야 하며, 어떤 상태는 화면 안에서만 잠깐 필요합니다. 이 구분이 흐려지면 사용자는 링크를 공유했는데 같은 화면이 열리지 않거나, 뒤로 가기를 눌렀는데 예상과 다른 위치로 이동하는 경험을 하게 됩니다.

URL 상태는 사용자가 "현재 보고 있는 관점"을 표현하는 데 적합합니다.

## 공유 가능한 상태는 URL에 둔다

검색어, 필터, 정렬, 페이지 번호처럼 같은 화면을 다른 사람에게 전달할 때 의미가 있는 값은 URL에 두는 편이 좋습니다.

```text
/issues?status=open&assignee=me&sort=updated_desc&page=2
```

이렇게 하면 새로고침, 북마크, 링크 공유, 브라우저 히스토리가 모두 자연스럽게 동작합니다. 반대로 이런 값을 컴포넌트 내부 상태에만 두면 사용자는 같은 조건을 다시 맞춰야 합니다.

## 임시 UI 상태는 로컬에 둔다

모든 상태를 URL에 넣으면 주소가 불필요하게 복잡해집니다. 드롭다운 열림 여부, hover 상태, 모달 내부 입력 중간값, 토스트 노출 여부처럼 순간적인 UI 상태는 로컬 상태가 더 적합합니다.

판단 기준은 간단합니다. 새로고침했을 때 유지되어야 하는 값인가, 뒤로 가기와 앞으로 가기로 복원되어야 하는 값인가, 링크로 공유할 의미가 있는 값인가를 묻습니다. 셋 중 하나라도 강하면 URL 후보입니다.

## 기본값은 URL에서 생략한다

URL 상태를 설계할 때는 기본값을 생략해야 주소가 읽기 쉬워집니다. 예를 들어 기본 정렬이 최신순이고 기본 페이지가 1이라면 다음처럼 만들 수 있습니다.

```text
/issues?status=open
```

`sort=updated_desc&page=1`까지 항상 넣으면 상태 비교와 캐시 키가 불필요하게 늘어납니다. 기본값은 파서에서 채우고, URL에는 사용자가 바꾼 값만 남기는 방식이 좋습니다.

## 파싱과 직렬화를 한곳에 모은다

쿼리 파라미터를 여러 컴포넌트에서 직접 읽고 쓰면 형식이 금방 흩어집니다. `page`를 어떤 곳은 문자열로, 어떤 곳은 숫자로 다루고, 존재하지 않는 필터 값을 처리하는 방식도 달라집니다.

URL 상태는 작은 어댑터로 모으는 편이 안전합니다.

```ts
type IssueQuery = {
  status: "open" | "closed" | "all";
  page: number;
  sort: "updated_desc" | "created_desc";
};
```

이 어댑터가 기본값, 허용 값, 숫자 변환, 잘못된 값 보정을 담당하면 화면 컴포넌트는 도메인 상태만 다룰 수 있습니다.

## 체크리스트

- 검색, 필터, 정렬, 페이지네이션은 URL 상태 후보로 본다.
- 공유할 의미가 없는 임시 UI 상태는 로컬에 둔다.
- 기본값은 URL에서 생략한다.
- 쿼리 파싱과 직렬화는 한곳에 모은다.
- 잘못된 쿼리 값은 깨진 화면 대신 안전한 기본값으로 보정한다.

좋은 URL 상태 설계는 사용자가 브라우저를 믿을 수 있게 만듭니다. 링크, 새로고침, 뒤로 가기가 자연스럽게 동작하면 화면 상태 관리의 절반은 이미 해결된 셈입니다.
