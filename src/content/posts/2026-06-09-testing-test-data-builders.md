테스트가 늘어날수록 데이터 준비 코드가 테스트의 절반을 차지하는 경우가 많습니다. 사용자, 조직, 주문, 권한, 구독 상태를 매번 직접 만들다 보면 정작 테스트가 검증하는 의도가 흐려집니다. 더 나쁜 문제는 도메인 필드가 하나 추가될 때 수십 개 fixture가 동시에 깨지는 것입니다.

Test data builder는 테스트 데이터의 기본값을 한곳에 두고, 각 테스트가 중요한 차이만 표현하게 해주는 패턴입니다.

## 기본적으로 유효한 객체를 만든다

Builder의 첫 번째 원칙은 아무 옵션 없이 호출해도 유효한 객체를 만드는 것입니다.

```ts
const user = userBuilder().build();
```

이 사용자는 저장 가능한 이메일, 활성 상태, 기본 권한처럼 테스트 대부분에서 필요한 정상 기본값을 가져야 합니다. 테스트가 관심 없는 필드까지 매번 채우게 만들면 중복이 늘어납니다.

## 차이는 메서드 이름으로 드러낸다

테스트에서 중요한 조건은 읽히는 이름으로 표현합니다.

```ts
const user = userBuilder()
  .withRole("admin")
  .withoutVerifiedEmail()
  .build();
```

이 코드는 raw object literal보다 의도가 분명합니다. 특히 권한, 구독, 상태 전이처럼 도메인 의미가 있는 값은 `status: "P"`보다 `withPendingSubscription()` 같은 이름이 리뷰하기 쉽습니다.

## 랜덤값은 조심해서 쓴다

Builder가 매번 랜덤 데이터를 만들면 충돌은 줄지만 재현성이 나빠질 수 있습니다. 테스트 실패 로그에 나온 값을 다시 만들 수 없으면 디버깅이 어려워집니다.

랜덤이 필요하다면 seed를 고정하거나, 실패 메시지에 생성된 값을 충분히 남깁니다. 유니크 이메일이 필요한 정도라면 단조 증가 카운터가 더 단순할 때도 많습니다.

## DB 저장과 순수 객체 생성을 분리한다

테스트에 따라 순수 객체만 필요한 경우와 실제 DB row가 필요한 경우가 다릅니다. 두 작업을 하나의 builder에 섞으면 테스트가 불필요하게 느려집니다.

```ts
userBuilder().build();
await userBuilder().create(db);
```

`build`는 메모리 객체를 만들고, `create`는 저장까지 담당하게 분리하면 단위 테스트와 통합 테스트의 비용을 조절하기 쉽습니다.

## 체크리스트

- builder는 기본적으로 유효한 데이터를 만든다.
- 테스트별 차이만 명시한다.
- 도메인 의미가 있는 상태는 메서드 이름으로 표현한다.
- 랜덤 데이터는 재현 가능성을 해치지 않게 제한한다.
- 순수 객체 생성과 DB 저장을 분리한다.

좋은 테스트 데이터는 테스트의 주장을 가리지 않습니다. Builder를 쓰면 데이터 준비 코드는 배경으로 물러나고, 테스트가 검증하려는 조건이 앞으로 나옵니다.
