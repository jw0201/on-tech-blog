Visual Regression Test는 화면이 의도치 않게 바뀌었는지 스크린샷 비교로 확인하는 테스트입니다. 버튼 색이 바뀌거나 레이아웃이 깨지는 문제를 빠르게 잡을 수 있어 디자인 시스템과 마케팅 페이지에서 특히 유용합니다.

하지만 화면 전체를 무작정 캡처하면 테스트가 너무 자주 깨집니다. 폰트 렌더링, 날짜, 애니메이션, 이미지 로딩, 브라우저 차이까지 모두 변경으로 잡히기 때문입니다. 시각 테스트는 신뢰할 영역과 무시할 영역을 분명히 해야 효과가 있습니다.

## 안정적인 컴포넌트부터 시작한다

처음부터 전체 앱 페이지를 스냅샷으로 묶으면 실패 원인을 찾기 어렵습니다. 버튼, 입력창, 모달, 카드처럼 독립적인 컴포넌트부터 시작하는 편이 좋습니다.

좋은 후보는 다음 조건을 만족합니다.

- 외부 데이터 의존성이 낮다.
- 애니메이션이 없거나 끌 수 있다.
- 상태 조합이 명확하다.
- 디자인 변경 영향이 크다.

디자인 시스템 컴포넌트는 시각 테스트의 투자 대비 효과가 높습니다. 한 컴포넌트의 변경이 여러 화면에 퍼지기 때문입니다.

## 동적인 값은 고정한다

시각 테스트에서 날짜, 시간, 랜덤 ID, 아바타 이미지, 광고 영역은 흔한 실패 원인입니다. 테스트 환경에서는 이런 값을 고정해야 합니다.

```text
현재 시간: 2026-06-14T09:00:00+09:00
랜덤 시드: fixed
네트워크 이미지: fixture로 대체
애니메이션: prefers-reduced-motion 적용
사용자 데이터: 테스트 계정 seed
```

동적 값을 고정하지 않으면 테스트 실패가 제품 변경 때문인지 환경 차이 때문인지 구분하기 어렵습니다.

## 픽셀 차이 허용치를 기능별로 다르게 둔다

모든 화면에 같은 임계값을 적용하면 문제가 생깁니다. 아이콘이나 버튼처럼 작은 컴포넌트는 1픽셀 차이도 중요할 수 있지만, 큰 대시보드 화면에서는 미세한 폰트 안티앨리어싱 차이를 허용해야 할 수 있습니다.

허용치는 낮을수록 좋은 것이 아니라 팀이 실제로 대응할 수 있을 만큼 현실적이어야 합니다. 매일 깨지는 테스트는 곧 무시됩니다.

## 의미 검증을 대체하지 않는다

스크린샷이 같다고 기능이 맞는 것은 아닙니다. 버튼이 보이지만 클릭이 안 될 수 있고, 접근성 이름이 잘못되었을 수 있습니다. Visual Regression Test는 DOM 기반 테스트, 접근성 테스트, E2E 테스트를 보완하는 역할입니다.

예를 들어 결제 버튼을 검증한다면 다음 테스트가 서로 다른 책임을 갖습니다.

- 단위 테스트: 가격 계산 함수가 맞는가
- 접근성 테스트: 버튼 이름과 포커스 이동이 맞는가
- E2E 테스트: 결제 요청이 성공하는가
- 시각 테스트: 버튼과 가격 영역이 의도치 않게 깨지지 않았는가

역할을 섞으면 어느 테스트도 선명하지 않습니다.

## 승인 흐름을 코드 리뷰에 붙인다

시각 변경이 의도된 것이라면 기준 이미지를 업데이트해야 합니다. 이 작업은 자동으로 몰래 처리되면 안 됩니다. 리뷰어가 변경 전후 이미지를 보고 승인해야 합니다.

PR 코멘트에 diff 이미지를 붙이고, 승인된 경우에만 baseline을 갱신하는 흐름이 좋습니다. 기준 이미지 변경도 코드 변경과 같은 리뷰 대상입니다.

## 체크리스트

- 전체 페이지보다 안정적인 컴포넌트부터 시작한다.
- 날짜, 랜덤, 이미지, 애니메이션을 테스트에서 고정한다.
- 화면 성격에 맞게 픽셀 차이 허용치를 조정한다.
- 기능 검증과 접근성 검증을 시각 테스트로 대체하지 않는다.
- baseline 업데이트는 리뷰 승인 흐름에 포함한다.

Visual Regression Test는 화면 품질을 지키는 강력한 안전망입니다. 다만 무엇을 비교할지, 무엇을 무시할지 정하지 않으면 안전망이 아니라 소음이 됩니다.
