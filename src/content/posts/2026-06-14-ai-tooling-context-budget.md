AI 코딩 도구는 많은 코드를 읽을수록 무조건 좋아지는 것처럼 보입니다. 하지만 실제로는 컨텍스트에도 예산이 있습니다. 관련 없는 파일, 오래된 논의, 중복 로그가 많이 들어가면 모델은 중요한 제약을 놓치고 엉뚱한 방향으로 확신할 수 있습니다.

좋은 결과를 얻으려면 "많이 넣기"보다 "문제 해결에 필요한 정보를 선별하기"가 중요합니다. 사람에게 코드 리뷰를 부탁할 때도 전체 저장소를 던지는 대신 변경 의도와 관련 파일을 알려주는 것과 같습니다.

## 목표를 한 문장으로 고정한다

에이전트가 가장 먼저 알아야 할 것은 작업의 성공 조건입니다. "이 버그 고쳐줘"보다 다음처럼 쓰면 훨씬 낫습니다.

```text
로그인 후 새로고침하면 세션은 유지되지만 프로필 API가 401을 반환하는 문제를 고쳐줘.
성공 기준은 새로고침 후 /me 요청이 200이고, 기존 로그인 테스트가 통과하는 것이다.
```

목표 문장에는 증상, 기대 동작, 검증 방법이 들어가야 합니다. 이 세 가지가 없으면 에이전트는 코드를 많이 읽고도 무엇을 끝으로 봐야 할지 헷갈립니다.

## 관련 파일 목록을 작게 시작한다

처음부터 저장소 전체를 읽게 하기보다 진입점을 알려주는 편이 좋습니다.

- 실패하는 테스트 파일
- 에러가 발생한 컴포넌트나 핸들러
- 최근 변경된 커밋
- 관련 설정 파일
- 재현 명령

이후 에이전트가 필요하면 검색해서 범위를 넓히게 하면 됩니다. 중요한 것은 처음 컨텍스트에 노이즈를 적게 넣는 것입니다.

## 로그는 원문 전체보다 핵심 구간을 준다

빌드 로그나 테스트 로그는 길어지기 쉽습니다. 전체 로그를 붙이면 중요한 줄이 묻힙니다. 에러 메시지, 스택 트레이스 상단, 실패한 테스트 이름, 재현 환경만 먼저 전달하는 것이 좋습니다.

```text
실패 테스트: auth/session-refresh.spec.ts
명령: npm test -- session-refresh
핵심 에러: Expected status 200, received 401
스택 상단: src/server/auth/session.ts:74
```

에이전트는 이 정보를 바탕으로 파일을 읽고, 필요할 때 전체 로그를 다시 확인할 수 있습니다.

## 설계 제약을 명시한다

모델은 보통 가능한 해법을 많이 떠올립니다. 하지만 제품에서는 모든 해법이 허용되지 않습니다. 예를 들어 다음 제약은 반드시 처음에 알려야 합니다.

- public API 응답 형식 변경 금지
- DB 마이그레이션 없이 해결
- 새 의존성 추가 금지
- 기존 디자인 시스템 컴포넌트만 사용
- 성능상 동기 호출 추가 금지

이런 제약을 뒤늦게 말하면 이미 작성된 코드가 크게 흔들립니다. 컨텍스트 예산을 아끼려면 금지 조건도 초반에 짧게 고정해야 합니다.

## 중간 요약을 갱신한다

긴 작업에서는 컨텍스트가 누적됩니다. 이때 오래된 가설과 이미 버린 접근이 계속 남아 있으면 품질이 떨어집니다. 중간에 다음 형식으로 상태를 정리하면 좋습니다.

```text
현재 확인된 사실:
- refresh token은 정상 발급된다.
- /me 요청에는 Authorization 헤더가 없다.
- 원인은 fetch wrapper가 쿠키 인증 경로를 bearer 인증으로만 처리하는 것이다.

남은 작업:
- fetch wrapper 수정
- 세션 새로고침 테스트 추가
- npm test 실행
```

요약은 에이전트뿐 아니라 사람에게도 좋습니다. 문제 해결의 방향이 맞는지 빠르게 확인할 수 있습니다.

## 체크리스트

- 작업 목표를 증상, 기대 동작, 검증 방법으로 쓴다.
- 관련 파일과 재현 명령을 작게 제공한다.
- 긴 로그는 실패 지점 중심으로 요약한다.
- API, DB, 의존성, 디자인 제약을 먼저 명시한다.
- 긴 작업에서는 확인된 사실과 남은 작업을 갱신한다.

AI 도구와의 협업 품질은 프롬프트의 길이가 아니라 컨텍스트의 밀도에서 나옵니다. 필요한 정보를 적절한 순서로 주는 것이 가장 강력한 생산성 향상입니다.
