폼 검증은 흔히 "사용자가 틀리게 입력했을 때 메시지를 보여주는 일"로 축소됩니다. 하지만 실제 서비스에서는 훨씬 더 넓은 문제입니다. 서버가 받는 데이터 형식, 저장 가능한 값의 범위, 비즈니스 규칙, 접근성, 국제화, 에러 복구 경험까지 모두 연결됩니다.

폼 검증을 UI 코드 안에 흩어 놓으면 같은 규칙이 여러 곳에서 다르게 구현됩니다. 회원가입 화면에서는 비밀번호가 8자 이상인데, 계정 설정 화면에서는 10자 이상으로 검증하는 식입니다. 이런 차이는 사용자에게는 버그이고, 운영자에게는 문의와 데이터 정합성 문제로 돌아옵니다.

## 서버 규칙이 기준이다

최종 검증 책임은 서버에 있습니다. 프론트엔드 검증은 빠른 피드백을 주기 위한 복제본일 뿐입니다. 따라서 폼 검증을 설계할 때는 먼저 서버가 받아들이는 입력 계약을 명확히 해야 합니다.

```ts
type SignupInput = {
  email: string;
  password: string;
  displayName: string;
  marketingConsent: boolean;
};
```

이 타입만으로는 충분하지 않습니다. 이메일은 어떤 형식을 허용하는지, 비밀번호 길이는 얼마인지, 이름에 공백을 허용하는지, 동의 값의 기본값은 무엇인지까지 같이 정의해야 합니다. 가능하면 서버와 클라이언트가 같은 스키마를 공유하거나, API 문서에서 검증 규칙을 생성할 수 있게 만드는 편이 좋습니다.

## 즉시 검증과 제출 검증을 나눈다

모든 검증을 입력 중에 즉시 보여주면 사용자가 방해받습니다. 특히 이메일, 전화번호, 날짜처럼 입력 중에는 불완전할 수밖에 없는 값은 조급한 에러 메시지가 피로감을 만듭니다.

실무에서는 검증 시점을 나누는 편이 안정적입니다.

- 필수 입력 여부는 blur 이후 또는 제출 시점에 보여준다.
- 글자 수 제한은 입력 중에도 카운터로 알려준다.
- 서버 확인이 필요한 중복 검사는 debounce와 취소 처리를 적용한다.
- 제출 버튼 비활성화만 믿지 말고 제출 시점에 다시 검증한다.

즉시 검증은 사용자를 돕는 신호여야지, 입력을 감시하는 장치처럼 느껴지면 안 됩니다.

## 에러 메시지는 필드 옆에 둔다

검증 실패를 상단 토스트 하나로만 보여주면 사용자는 어떤 필드를 고쳐야 하는지 찾아야 합니다. 필드 단위 에러는 해당 입력 바로 아래에 배치하고, 제출 실패 후에는 첫 번째 에러 필드로 포커스를 이동시키는 편이 좋습니다.

```html
<input aria-invalid="true" aria-describedby="email-error" />
<p id="email-error">사용할 수 있는 이메일 주소를 입력해 주세요.</p>
```

접근성 속성도 중요합니다. 시각적으로 빨간 테두리를 보여주는 것만으로는 스크린 리더 사용자에게 충분하지 않습니다. `aria-invalid`, `aria-describedby`, 명확한 레이블을 함께 사용해야 합니다.

## 서버 에러를 필드 에러로 매핑한다

서버가 `400 Bad Request`만 반환하고 메시지 문자열 하나를 내려주면 프론트엔드는 에러를 정확한 필드에 붙이기 어렵습니다. API는 검증 실패를 구조화해서 반환하는 편이 좋습니다.

```json
{
  "errors": [
    { "field": "email", "code": "already_taken" },
    { "field": "password", "code": "too_short" }
  ]
}
```

프론트엔드는 `code`를 사용자 언어에 맞는 문구로 변환합니다. 이렇게 하면 서버는 안정적인 에러 코드를 제공하고, 클라이언트는 화면 맥락에 맞게 메시지를 조정할 수 있습니다.

## 체크리스트

- 서버 검증 규칙을 기준으로 프론트엔드 검증을 맞춘다.
- 입력 중 검증, blur 검증, 제출 검증의 역할을 분리한다.
- 필드 단위 에러와 접근성 속성을 함께 설계한다.
- 서버 검증 실패는 구조화된 필드 에러로 반환한다.
- 제출 버튼 비활성화와 별개로 제출 시점 검증을 반드시 수행한다.

좋은 폼 검증은 사용자를 막는 코드가 아니라 올바른 제출까지 안내하는 흐름입니다. 검증 규칙을 계약으로 다루면 프론트엔드와 백엔드가 같은 언어로 입력 데이터를 관리할 수 있습니다.
