운영 알림이 많다고 안정적인 것은 아닙니다. CPU 80%, 에러 10건, 지연 500ms 같은 단순 임계값은 상황을 너무 많이 놓치거나 너무 자주 울립니다. 중요한 것은 사용자가 약속받은 신뢰성을 얼마나 빠르게 잃고 있는지입니다.

SLO Burn Rate 알림은 에러 예산이 소모되는 속도를 기준으로 문제를 감지합니다. "지금 속도라면 한 달치 허용 실패를 몇 시간 만에 다 써버리는가"를 보는 방식입니다.

## SLO와 에러 예산을 먼저 정한다

Burn Rate를 쓰려면 먼저 SLO가 있어야 합니다. 예를 들어 API 성공률 SLO를 99.9%로 정하면 한 달 요청 중 0.1%는 실패할 수 있습니다. 이 0.1%가 에러 예산입니다.

```text
SLO: 30일 동안 checkout API 성공률 99.9%
에러 예산: 전체 요청의 0.1%
실패 정의: 5xx, timeout, 2초 초과 응답
```

실패 정의가 중요합니다. 단순히 5xx만 보면 사용자가 실제로 겪는 느린 응답을 놓칠 수 있습니다.

## Burn Rate는 소모 속도다

Burn Rate가 1이면 에러 예산을 기간에 맞춰 정상 속도로 쓰고 있다는 뜻입니다. Burn Rate가 10이면 같은 예산을 10배 빠르게 쓰고 있다는 뜻입니다.

예를 들어 30일 SLO에서 Burn Rate 14.4가 1시간 지속되면 하루치에 가까운 예산을 1시간 만에 써버리는 셈입니다. 이런 상황은 즉시 대응할 가치가 있습니다.

## 짧은 창과 긴 창을 함께 본다

짧은 시간 창만 보면 순간적인 스파이크에 너무 민감합니다. 긴 시간 창만 보면 급격한 장애를 늦게 봅니다. 그래서 보통 두 개의 창을 함께 사용합니다.

```text
긴급 알림: 5분 burn rate와 1시간 burn rate가 모두 높음
티켓 알림: 30분 burn rate와 6시간 burn rate가 모두 높음
```

짧은 창은 빠른 감지를, 긴 창은 지속성을 확인합니다. 두 조건이 함께 만족될 때 알리면 소음을 줄일 수 있습니다.

## 사용자 여정별로 나눈다

전체 API 성공률만 보면 중요한 장애가 희석될 수 있습니다. 체크아웃, 로그인, 파일 업로드처럼 사용자 가치가 큰 경로는 별도 SLO를 가져야 합니다.

```text
checkout_success_rate
login_success_rate
search_latency_under_1s
webhook_delivery_success_rate
```

내부 배치 API 실패와 결제 API 실패를 같은 알림으로 보면 우선순위를 잘못 판단할 수 있습니다.

## 알림에는 행동을 붙인다

좋은 Burn Rate 알림은 숫자만 보내지 않습니다. 어떤 SLO가, 어느 기간 동안, 얼마나 빠르게 예산을 쓰고 있으며, 첫 조사 링크가 무엇인지 포함해야 합니다.

```text
checkout API SLO burn rate 18x
window: 5m and 1h
top errors: PAYMENT_UPSTREAM_TIMEOUT, DB_POOL_EXHAUSTED
runbook: /runbooks/checkout-slo
dashboard: /dashboards/checkout
```

알림을 받은 사람이 바로 다음 행동을 시작할 수 있어야 합니다.

## 체크리스트

- 사용자 관점의 SLO와 실패 정의를 먼저 정한다.
- 에러 예산 대비 소모 속도로 Burn Rate를 계산한다.
- 짧은 창과 긴 창을 함께 조건으로 사용한다.
- 핵심 사용자 여정은 별도 SLO로 분리한다.
- 알림에 원인 후보, 대시보드, 런북 링크를 포함한다.
- 단순 인프라 임계값 알림은 SLO 알림을 보완하는 용도로 둔다.

SLO Burn Rate는 알림을 줄이기 위한 기술이 아니라 중요한 알림만 남기기 위한 운영 방식입니다. 사용자가 실제로 잃는 신뢰성을 기준으로 대응하면 팀의 집중력이 좋아집니다.
