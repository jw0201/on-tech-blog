AI 코딩 에이전트는 빠르게 코드를 바꿀 수 있지만, 빠르다는 사실이 곧 안전하다는 뜻은 아닙니다. 실무에서 중요한 것은 "AI가 코드를 얼마나 많이 쓰는가"보다 "변경을 어떤 루프로 검증하는가"입니다.

좋은 루프는 작게 지시하고, 근거를 확인하고, 테스트로 고정하고, 사람이 마지막 결정을 내리는 흐름입니다.

## 작업 단위를 작게 자른다

AI에게 "결제 시스템을 개선해줘"라고 맡기면 변경 범위가 너무 넓어집니다. 대신 하나의 관찰 가능한 결과로 쪼개야 합니다.

- 실패하는 테스트 하나를 통과시킨다.
- 특정 API의 validation 누락을 보완한다.
- 한 컴포넌트의 접근성 경고를 해결한다.
- 오래된 helper 하나를 새 인터페이스에 맞춘다.

작은 단위일수록 diff를 읽기 쉽고, AI의 추론이 빗나갔을 때 되돌리기 쉽습니다.

## 먼저 코드를 읽게 한다

새 기능을 바로 만들게 하기보다 관련 파일, 테스트, 호출 경로를 먼저 확인하게 하는 편이 좋습니다. 에이전트가 기존 패턴을 읽지 않고 코드를 작성하면 프로젝트 안에 두 번째 스타일이 생깁니다.

요청은 이렇게 구체적이면 좋습니다.

```text
이 변경을 하기 전에 관련 라우트, 서비스, 테스트 파일을 먼저 읽고
현재 패턴을 요약한 뒤 그 패턴 안에서 수정해줘.
```

요약이 틀렸다면 아직 구현을 맡길 단계가 아닙니다. 읽기 결과부터 바로잡아야 합니다.

## 테스트를 변경의 계약으로 둔다

AI가 만든 코드는 문법상 그럴듯하지만 경계 조건을 놓치기 쉽습니다. 그래서 변경 요청에는 테스트 기대치를 같이 넣는 것이 좋습니다.

- 기존 테스트가 깨지면 원인을 설명한다.
- 버그 수정이면 회귀 테스트를 추가한다.
- 리팩터링이면 테스트 결과가 그대로임을 확인한다.
- 테스트를 못 돌렸다면 이유를 남긴다.

테스트가 없는 코드라면 최소한 작은 characterization test를 추가하는 것이 낫습니다. 기존 동작을 고정한 뒤 바꾸면 리뷰가 훨씬 쉬워집니다.

## 리뷰에서는 생성 의도보다 diff를 본다

AI가 "안전하게 처리했다"고 설명해도 리뷰 기준은 diff입니다. 특히 다음 항목을 집중해서 봅니다.

- 기존 권한 검사나 validation이 빠지지 않았는가
- 에러 처리가 호출자 계약과 맞는가
- 새 dependency가 꼭 필요한가
- logging에 민감 정보가 들어가지 않았는가
- 동시성, 재시도, 캐시 무효화가 바뀌지 않았는가
- 테스트가 실제 실패를 잡는 형태인가

AI 설명은 리뷰의 보조 자료일 뿐입니다. 최종 판단은 코드와 테스트 결과로 해야 합니다.

## 한 번에 포맷과 기능을 섞지 않는다

AI는 종종 주변 코드를 같이 정리하려고 합니다. 하지만 포맷 변경과 기능 변경이 섞이면 리뷰 비용이 커집니다. 먼저 기능 diff를 작게 만들고, 필요하면 별도 커밋으로 포맷을 정리하는 편이 낫습니다.

프롬프트에도 범위를 명시합니다.

```text
요청한 버그 수정에 필요한 파일만 바꿔줘.
대규모 포맷팅, 이름 변경, unrelated refactor는 하지 마.
```

## 에이전트에게 리뷰어 역할도 맡긴다

구현을 맡긴 뒤 같은 에이전트에게 바로 "괜찮지?"라고 물으면 긍정 편향이 생길 수 있습니다. 대신 다른 관점의 리뷰 프롬프트를 사용합니다.

```text
방금 변경을 코드 리뷰어 관점에서 봐줘.
버그, 보안 위험, 깨질 수 있는 기존 동작, 빠진 테스트만 찾아줘.
칭찬이나 요약은 뒤로 미뤄줘.
```

이 리뷰 결과도 그대로 믿지는 말고 사람이 확인해야 합니다. 그래도 놓친 케이스를 찾는 데 꽤 도움이 됩니다.

## 체크리스트

- 작업을 한 번에 리뷰 가능한 크기로 제한한다.
- 구현 전에 관련 코드와 테스트를 읽게 한다.
- 변경마다 테스트 실행 결과를 남긴다.
- 설명보다 diff와 테스트를 우선한다.
- 포맷 변경과 기능 변경을 분리한다.
- AI 리뷰는 보조 수단으로 쓰고 최종 책임은 사람이 진다.

AI 코딩 에이전트는 팀의 손을 빠르게 해 주지만, 팀의 판단을 대신하지는 못합니다. 좋은 사용법은 더 많은 코드를 생성하는 것이 아니라 더 짧은 검증 루프를 만드는 것입니다.
